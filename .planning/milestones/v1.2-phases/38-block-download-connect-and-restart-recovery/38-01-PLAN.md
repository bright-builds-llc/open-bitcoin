# Plan 38-01: Durable Block Reconciliation And Replay-Safe Chain Advancement

## Objective

Extend the daemon sync runtime from Phase 37’s durable header-first state into a bounded block-download and reconnect pipeline that can recover after restart, reconnect already-persisted blocks, and use replay-safe chainstate transitions when the durable best header branch overtakes the current active chain.

## Requirements

- SYNCMAIN-03: The daemon sync task must keep bounded restart-safe runtime behavior.
- CHAINMAIN-02: Missing blocks for the best durable header chain must be requested, downloaded, validated, persisted, and connected.
- CHAINMAIN-03: Restart recovery must reuse durable headers and stored blocks instead of replaying everything from scratch.
- CHAINMAIN-04: Branch takeovers must use durable undo or replay-safe transitions.
- CHAINMAIN-05: The runtime must keep block-download work bounded and surface actionable insufficient-resource guidance.
- RESUME-01: Recovery must reconstruct sync work from durable headers, chainstate, and runtime metadata.
- RESUME-02: Invalid data and storage/runtime failures must stay typed and operator visible.
- RESUME-03: Corrupt or incompatible durable state must point operators toward restart/repair/reindex guidance without implicit mutation.
- VERMAIN-01: Repo-native verification must stay hermetic.
- VERMAIN-02: Deterministic tests must cover restart, competing branches, and bounded request behavior.

## Threat Model

| Threat | Severity | Mitigation |
| --- | --- | --- |
| Durable headers advance but block sync stalls forever after restart | High | Runtime-owned request planner issues bounded `getdata` from durable best-chain state even with no fresh headers. |
| Persisted blocks are replayed unsafely or connected twice | High | Reconciliation derives work from current chainstate plus durable header chain and treats already-connected blocks as idempotent. |
| Better branch blocks are downloaded but never activate | High | Reuse undo-backed `reorg` with deterministic header-store best-tip selection. |
| Resource pressure causes unbounded requests or opaque failures | Medium | Add explicit per-peer/global caps and typed operator guidance for insufficient safe progress. |
