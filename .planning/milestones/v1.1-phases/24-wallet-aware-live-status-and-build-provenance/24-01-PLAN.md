# Phase 24 Plan 01: Make Live Status Wallet-Aware

## Objective

Repair the shared live status path so wallet-scope ambiguity and missing-wallet
conditions degrade only wallet fields, while selected or sole wallets still use
wallet-scoped RPC routing when local evidence supports it.

## Scope

- Wallet route resolution for status and dashboard runtime setup
- Wallet-aware status RPC client routing and error parsing
- Shared status collector behavior for wallet-only failures
- Focused tests for selected-wallet, multiwallet, and wallet-failure paths

## Tasks

1. Add a typed wallet-access model for the status runtime so root access,
   selected-wallet access, and locally-known ambiguity stay explicit.
2. Route status wallet RPC calls through `/wallet/<name>` when the local
   registry exposes a selected or sole wallet.
3. Parse JSON-RPC error payloads and keep wallet failures from collapsing the
   node state to unreachable.
4. Add focused tests for:
   - wallet-only failure while node RPC stays healthy
   - selected-wallet routing
   - ambiguous multiwallet registry handling
   - regression coverage for the fake-running operator binary path
