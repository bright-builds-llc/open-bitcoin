# Phase 18: Service Lifecycle Integration - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning
**Mode:** Yolo

<domain>
## Phase Boundary

Implement macOS launchd and Linux systemd service lifecycle commands — install, uninstall, enable, disable, and status — wiring the already-stubbed `ServiceCommand` enum and `ServiceStatus` model into real behavior with dry-run safety, isolation-safe tests, and privilege transparency. This phase does not implement the Ratatui dashboard, wallet expansion, migration wizard, or external observability export.

</domain>

<decisions>
## Implementation Decisions

### Service Module Structure
- **D-01:** Place service lifecycle logic in a new `operator/service/` submodule within `open-bitcoin-cli`. The submodule defines a `ServiceManager` trait with methods for install, uninstall, enable, disable, and status inspection. Platform-specific adapters (LaunchdAdapter, SystemdAdapter) implement the trait. A `FakeServiceManager` implements the trait for tests.
- **D-02:** Pure functions — plist content generation, unit file content generation, service state classification — live in `operator/service/` without filesystem or subprocess side effects. Effectful adapter methods (writing files, invoking `launchctl`/`systemctl`) live in platform adapter implementations.
- **D-03:** Wire `OperatorCommand::Service` in `operator/runtime.rs` to dispatch to the new service module instead of returning the current "deferred to Phase 18" error.

### Plist and Unit File Generation
- **D-04:** Generate plist and systemd unit file contents from typed Rust structs using pure string/serialization methods — no third-party template engines. Each generator exposes a `dry_run_content() -> String` method so the generated file body can be surfaced to operators before writes.
- **D-05:** Plist output targets macOS LaunchAgents (`~/Library/LaunchAgents/`). Unit file output targets systemd user scope (`~/.config/systemd/user/`). File names follow the project service name convention; use `org.open-bitcoin.node.plist` and `open-bitcoin-node.service`.
- **D-06:** Generated content includes: daemon binary path (resolved via `std::env::current_exe()` or config-injected path), data directory, config path, log path, recovery behavior (`KeepAlive true` / `Restart=on-failure`), and human-readable comment header noting it was generated by Open Bitcoin.

### Platform Detection
- **D-07:** Use `#[cfg(target_os = "macos")]` / `#[cfg(target_os = "linux")]` to select the active platform adapter at compile time. At runtime, the selected adapter is returned by a `platform_service_manager()` factory. For tests, the factory is replaced by injecting a `FakeServiceManager` through the `ServiceManager` trait object.
- **D-08:** If a service command is invoked on an unsupported platform, return a typed `ServiceError::UnsupportedPlatform` with a human-readable explanation rather than panicking or silently succeeding.

### Service Scope (User vs System)
- **D-09:** Default to user-level service scope: LaunchAgents on macOS, `systemctl --user` on Linux. Surface the scope prominently in dry-run output and status. This avoids requiring root for typical operator use and keeps service files under the user home directory.
- **D-10:** Do not add a system-level scope option in this phase. System-level service support is deferred — the current interface should make it straightforward to add a `--system` flag later without breaking user-level behavior.

### Dry-Run Safety
- **D-11:** Every install and uninstall command is dry-run by default. Operators see the generated file contents, target paths, and shell commands that would run before any write occurs. No filesystem or process state is mutated without an explicit `--apply` flag.
- **D-12:** Enable and disable commands print the `launchctl`/`systemctl` invocations that would run before executing them, but do not require `--apply` since they do not write files. They do surface privilege requirements and target scope in output.
- **D-13:** Privilege requirements (e.g., sudo needed for system scope, no sudo needed for user scope) are surfaced in the dry-run/status output before any action.

### Status Snapshot Integration
- **D-14:** Service status inspection (`open-bitcoin service status`) classifies the current state into: unmanaged (no service definition found), installed (plist/unit file present), enabled (service registered to start), running, failed, or stopped. Map these to the existing `ServiceStatus` struct fields (`manager`, `installed`, `enabled`, `running`) using `FieldAvailability<T>`.
- **D-15:** The `collect_status_snapshot` function in `operator/status.rs` already accepts `ServiceStatus`. Wire service state collection as an optional injected adapter so `open-bitcoin status` can populate `service.*` fields when service inspection succeeds, and report `unavailable("service manager not inspected")` when not wired or when collection fails.
- **D-16:** `open-bitcoin service status` output displays: manager (launchd/systemd), service file path, enabled state, running state, recent manager-diagnostic output if available, and links to relevant log paths.

### Testing
- **D-17:** Tests use a `FakeServiceManager` that records calls and returns deterministic outcomes. No test invokes `launchctl` or `systemctl`. Plist and unit file content generators are tested as pure functions with string snapshot assertions.
- **D-18:** Integration-level tests write to isolated temp directories and verify file contents and state transitions without touching real `~/Library/LaunchAgents/` or `~/.config/systemd/user/`.
- **D-19:** At least one test per command (install, uninstall, enable, disable, status) covers dry-run vs apply distinction, privilege-requirement surfacing, and typed error paths (missing binary, unsupported platform, already-installed guard).

### Claude's Discretion
- Exact naming of adapter structs, internal helper names, and field ordering are discretionary provided the contracts above are met.
- The `--apply` flag may be named `--apply` or `--execute` — choose what reads most naturally in help text alongside the dry-run description.
- Unit file `[Install]` section target (e.g., `default.target`) is discretionary; use a sensible systemd default.
- If a guard against double-install is needed (service file already present), return a typed error rather than silently overwriting. The `--force` or `--reinstall` path is discretionary but should be typed and documented.

</decisions>

<specifics>
## Specific Ideas

- Use `org.open-bitcoin.node.plist` for the macOS launchd service label and `open-bitcoin-node.service` for the systemd unit filename — consistent with project naming and distinguishable from Core/Knots service files.
- Dry-run output should read like a preview: show exactly which files would be written to which paths, and which commands would be run. Operators should not have to read source code to understand what `--apply` will do.
- When `open-bitcoin service status` detects that no service file exists, output should say "unmanaged — run `open-bitcoin service install --dry-run` to see what would be created" rather than just "not installed."
- Keep service status output quiet and information-dense in the same style as Phase 17 status output.

</specifics>
