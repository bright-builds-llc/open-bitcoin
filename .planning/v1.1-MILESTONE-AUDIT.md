# Milestone v1.1 Audit

## Verdict

Milestone `v1.1` does not meet its definition of done yet.

The audit found two kinds of blockers:

- audit evidence blockers: nine milestone requirements are orphaned from all
  v1.1 verification reports (`DB-01` through `DB-05`, `SYNC-01` through
  `SYNC-04`), which the workflow treats as unsatisfied even though the
  requirements ledger and summary frontmatter claim completion.

  unreachable when wallet RPC selection fails at the root endpoint.

The milestone also has seventeen partial requirements where verification
evidence exists but `requirements-completed` summary evidence and/or
`REQUIREMENTS.md` bookkeeping were not reconciled.

## Cross-Phase Integration

### Blocking Issues

| ID | Affected requirements | Finding | Evidence |
| --- | --- | --- | --- |
| `INT-01` | `SVC-01`, `SVC-02`, `SVC-03`, `DASH-03` | `open-bitcoin service install --apply` is incomplete on both macOS and Linux. The install path writes the plist or unit file and returns the commands that would need to run, but does not actually bootstrap, reload, or enable the service as part of the apply path. The dashboard install action inherits the same runtime path. | `packages/open-bitcoin-cli/src/operator/service/launchd.rs`, `packages/open-bitcoin-cli/src/operator/service/systemd.rs`, `packages/open-bitcoin-cli/src/operator/dashboard/action.rs` |
| `INT-02` | `OBS-01`, `OBS-02`, `DASH-01`, `WAL-05` | The live status/dashboard path treats wallet-scope RPC failures as node reachability failures. A healthy node with zero loaded wallets or multiple loaded wallets can render as `Daemon: unreachable` because `getwalletinfo` and `getbalances` are always called at the root RPC endpoint and any failure collapses the whole snapshot to `NodeRuntimeState::Unreachable`. | `packages/open-bitcoin-cli/src/operator/status/http.rs`, `packages/open-bitcoin-cli/src/operator/status.rs`, `packages/open-bitcoin-rpc/src/context/wallet_state.rs` |

## Broken Flows

| Flow | Break point | Impact |
| --- | --- | --- |
| `open-bitcoin service install --apply` | Apply mode stops after file write. | The service file is created, but the same command does not actually bootstrap or reload/enable the service. |
| Dashboard install action | Reuses the same service-install path. | Dashboard action confirms an install, then inherits the incomplete service apply behavior. |
| `open-bitcoin status` / `open-bitcoin dashboard` on nodes with zero or multiple wallets | Wallet RPC scope failure is treated as node unreachable. | Healthy nodes can look down or unreachable even when only wallet selection is ambiguous. |
| `open-bitcoin migrate plan --source-datadir <custom-path>` | Selected source must already be in detected installations. | Custom-location source installs degrade to manual review instead of a concrete plan. |

## Non-Blocking Deferred Risks

- Phase 19 still depends on live operator use to validate TUI ergonomics beyond
  snapshot tests.
- Phase 21 intentionally keeps migration apply-mode out of scope; future cutover
  work must preserve the dry-run-first safety posture.
- Phase 22's operator-runtime benchmark coverage is not fully end-to-end for
  status/dashboard collection: it benchmarks rendering and projection against
  `sample_status_snapshot()` fixtures rather than the real runtime collection
  path.
- Phase 22 keeps benchmark timing thresholds review-only rather than
  release-gating.
- Packaged/signed release installation, Windows service support, hosted
  dashboards, and migration apply mode remain intentionally deferred and are
  correctly non-claimed in the parity ledger.

     custom installs can be selected directly.
