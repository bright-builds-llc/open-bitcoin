# Phase 07.4: Let-else opportunity sweep - Research

**Researched:** 2026-04-19  
**Domain:** Final production-only `let...else` sweep after Phase 07.3  
**Confidence:** Medium

## Audit Outcome

| File | Decision | Why |
| --- | --- | --- |
| `packages/open-bitcoin-network/src/header_store.rs` | Defer | The remaining `if let` and tuple-construction branches already read clearly; a forced `let...else` rewrite adds indirection without reducing ambiguity. |
| `packages/open-bitcoin-node/src/network.rs` | Adopt | `collect_outbound` and the `serve_inventory` branches have clear continue-on-miss paths that become more direct with `let...else`. |
| `packages/open-bitcoin-wallet/src/wallet/build.rs` | Adopt | The optional change-output branch is long enough that a tiny helper with `let...else` makes the happy path easier to scan. |
| `packages/open-bitcoin-consensus/src/classify.rs` | Adopt | The `maybe_push_len` loop branch becomes simpler as a `let...else { continue; }` guard without changing parsing behavior. |

## Default Non-Targets

- `packages/open-bitcoin-wallet/src/descriptor.rs` — current parser-style
  `if let` chains represent mutually exclusive descriptor forms and are clearer
  left as-is.
- `packages/open-bitcoin-wallet/src/address.rs` — still overlaps the deferred
  panic or illegal-state sweep more than a pure `let...else` cleanup.
- All test-only files — out of scope for this phase unless a touched production
  path forces a narrow regression edit.

## Implementation Notes

- Keep edits local and behavior-neutral.
- Prefer one tiny private helper over a long inline `if let` expression only
  when it directly supports the `let...else` cleanup.
- Preserve signatures, ordering, reject reasons, and module boundaries.
- End on `bash scripts/verify.sh`.
