# Plan 44-01: Validation-Gated Peer Contribution Evidence

<objective>
Implement peer contribution attribution for PEER-03 so sync telemetry credits
only peers that contributed validated headers or accepted/preserved blocks,
while idle, waiting, stalled, and failed peers remain diagnosable without being
reported as useful progress.

Purpose: Phase 44 makes per-peer sync progress truthful before later v1.3 work
uses these surfaces for resource-bound hardening, support evidence, and public
mainnet progress closeout.

Output: Validation-gated runtime counters, deterministic regressions,
live-smoke peer contribution evidence, and operator documentation.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public peer input -> sync validation | Untrusted peers can send invalid headers, invalid blocks, stalls, or disconnects. |
| Sync runtime -> durable peer telemetry | Runtime-owned outcomes become operator status, logs, and live-smoke evidence. |
| Durable status -> support evidence | Local status JSON is normalized into human-readable Markdown and JSON reports. |
| Operator docs -> public-network troubleshooting | Documentation guides opt-in mainnet smoke interpretation without becoming default verification. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-01-01 | Tampering | Header/block contribution counters | mitigate | Count headers only after `receive_sync_message` succeeds and blocks only after `save_block` succeeds; invalid data updates activity/failure only. |
| T-44-01-02 | Spoofing | Peer usefulness in status/reporting | mitigate | Keep runtime-owned `PeerSyncOutcome` states and contribution counters as the single source for status and live-smoke rows; do not derive contribution from endpoint reachability. |
| T-44-01-03 | Repudiation | Failed connected peer diagnosis | mitigate | Preserve last activity, failure reason, and error on failed connected peer outcomes so invalid-data peers do not look never-active. |
| T-44-01-04 | Denial of Service | Peer attribution accounting | mitigate | Reuse existing bounded message, peer, and retry loops; do not add peer scoring, eviction heuristics, retention expansion, or public-network default checks. |
| T-44-01-05 | Information Disclosure | Live-smoke report rows | mitigate | Report only existing operator-visible peer endpoints, states, contribution counts, failure reason, and error; do not include datadir secrets, RPC credentials, or raw peer payloads. |
</threat_model>
