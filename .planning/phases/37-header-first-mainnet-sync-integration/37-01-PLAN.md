# Plan 37-01: Durable Header Continuation And Validation

## Objective

Turn the Phase 36 peer layer into a durable header-first sync pipeline that keeps batching headers, validates them contextually, preserves restart progress, and reports header-versus-block truth without starting block download yet.

## Requirements

- SYNCMAIN-03: The daemon-owned sync task must preserve coherent restart semantics while header sync advances.
- CHAINMAIN-01: Header-first sync must move from genesis, checkpoint, or durable tip to the best validated header chain.
- CHAINMAIN-03: Verified header progress must survive restart without needless replay.
- RESUME-01: Restart recovery must rebuild header sync state from durable snapshots.
- RESUME-02: Invalid headers and malformed peer data must stay typed and operator-visible.
- VERMAIN-01: Default verification must remain hermetic.
- VERMAIN-02: Deterministic tests must cover happy path, restart, invalid data, disconnects, and competing branches.

## Threat Model

| Threat | Severity | Mitigation |
| --- | --- | --- |
| Header sync silently stalls after one batch | High | Continue `getheaders` while new headers arrive and test multi-batch scripted peers. |
| Invalid headers enter durable state | High | Contextually validate headers before insertion and return typed invalid-data errors. |
| Header progress overstates chainstate progress | High | Keep separate header and block heights in summaries and status projections. |
| Restart throws away imported headers | High | Reload the durable header snapshot on open and assert restart reuse in tests. |
