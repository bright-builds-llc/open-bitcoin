# Phase 07.6: Enforce coinbase subsidy-plus-fees limits on the consensus and active chainstate paths - Context

**Gathered:** 2026-04-22

<domain>
## Phase Boundary

Phase 07.6 is a narrow consensus follow-on inserted after the still-open
Phase 07.5 parity sequence. It does not widen into a general chainstate or
node-hardening phase. It only:

1. enforces the Knots-style coinbase subsidy-plus-fees reward limit on the
   pure-core block-validation path,
2. enforces the same `bad-cb-amount` rule on the active chainstate
   block-connect path before live state is committed,
3. adds the focused proof chain needed to close this specific coinbase-limit
   acceptance gap without reopening unrelated `07.5` warnings or Phase 8
   interface work.

</domain>

<decisions>
## Implementation Decisions

### Scope and supporting-file boundary
- **D-01:** Limit the phase to the coinbase subsidy-plus-fees enforcement gap
  surfaced by `.planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-REVIEW.md`
  and the latest `07.5-VERIFICATION.md`.
- **D-02:** Allow only the minimal supporting-file surface needed for that fix:
  `packages/open-bitcoin-consensus/src/context.rs`,
  `packages/open-bitcoin-consensus/src/transaction.rs`,
  `packages/open-bitcoin-consensus/src/block.rs`,
  `packages/open-bitcoin-chainstate/src/engine.rs`, and focused consensus or
  chainstate regression files. Do not widen into unrelated `07.5` warnings
  about local-connect time handling or transactional reorg safety.

### Reward-limit contract
- **D-03:** Treat Knots `GetBlockSubsidy()` and the contextual coinbase payout
  limit in `validation.cpp` as the canonical behavior source. The Open Bitcoin
  path must reject overpaying coinbases with `bad-cb-amount` when
  `coinbase_outputs > subsidy(height) + total_fees`.
- **D-04:** Aggregate non-coinbase fees exactly once from the existing
  `validate_transaction_with_context()` path instead of recomputing them from
  outputs or introducing a second fee-calculation path.
- **D-05:** The active chainstate connect path must perform the reward-limit
  check before committing `self.utxos`, `self.undo_by_block`, or
  `self.active_chain`, so a failing coinbase leaves live state unchanged.

### Verification and closeout
- **D-06:** Add focused regressions for both the pure-core helper and the live
  chainstate path. The proof chain should show an exact-fee coinbase succeeds,
  a `+1` overpay fails with `bad-cb-amount`, and the rejected live-path attempt
  does not mutate the chainstate snapshot.
- **D-07:** The phase is not complete until the targeted reward-limit
  regressions pass, `bash scripts/verify.sh` passes, and the final summary names
  the new proof strings without reopening already-green `07.5` parity evidence.

### the agent's Discretion
- Exact helper names, whether subsidy helpers live in `block.rs` or a small
  sibling module, and how to express overflow diagnostics remain at the agent's
  discretion as long as the resulting path stays typed, narrow, and
  behavior-aligned with Knots.
- The planner or executor may reuse existing `Amount` arithmetic helpers rather
  than inventing new primitive math, provided the overflow behavior remains
  explicit and testable.

</decisions>

<specifics>
## Specific Ideas

- The triggering issue is that the current block-connect path can accept an
  overpaying coinbase because it computes per-transaction fees but never
  aggregates them into a `bad-cb-amount` check.
- Prefer a shared pure helper for coinbase payout validation so the same rule is
  used by both `validate_block_with_context()` and the live chainstate connect
  path.
- Use a zero-subsidy or tiny-halving-interval test fixture when it makes the
  exact-fee vs `+1` overpay boundary easier to express deterministically.
- Keep `07.5`’s already-green explicit-time, retarget, and lax-DER proof chain
  intact by reference rather than rerunning or rewriting it unless the new
  reward-limit proof chain specifically depends on it.

</specifics>
