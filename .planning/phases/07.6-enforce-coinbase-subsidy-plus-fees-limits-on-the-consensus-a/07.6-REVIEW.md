The phase covers the right high-level rule, but there are still two correctness gaps in the consensus math. One is consensus-critical in `check_tx_inputs`; the other misses the Knots `MoneyRange(nFees)` boundary and only starts failing much later at `i64` overflow.

## Critical Issues

### CR-01: `check_tx_inputs` can wrap input totals back into a valid satoshi range

**File:** `packages/open-bitcoin-consensus/src/context.rs:343-365`
**Issue:** `total_input_value` is updated with plain `+=` before the range check runs. That means large contextual transactions can overflow the `i64` accumulator before `"bad-txns-inputvalues-outofrange"` is returned. This is not just a debug-only panic risk: with 8,785 `MAX_MONEY` inputs, the wrapped signed total becomes `1_755_926_290_448_384`, which is still inside `0..=MAX_MONEY` and therefore can pass the current check in release builds. Because `validate_transaction_with_context` calls this helper directly, a caller that provides such a context can misclassify or accept an invalid transaction instead of rejecting it deterministically.
**Fix:**
```rust
let value = input_context.spent_output.value.to_sats();
let next_total_input_value = total_input_value
    .checked_add(value)
    .filter(|sum| (0..=MAX_MONEY).contains(sum))
    .ok_or_else(|| {
        tx_error(
            TxValidationResult::Consensus,
            "bad-txns-inputvalues-outofrange",
            None,
        )
    })?;

total_input_value = next_total_input_value;
```

Add a regression next to the existing overflow coverage in `packages/open-bitcoin-consensus/src/context/tests.rs` that builds the smallest wrapping case instead of only testing sums that stay below `i64::MAX`.

## Warnings

### WR-01: Accumulated block fees are only bounded by `i64`, not by `MAX_MONEY`

**File:** `packages/open-bitcoin-consensus/src/block.rs:310-325`, `packages/open-bitcoin-chainstate/src/engine.rs:117-145`
**Issue:** Both fee accumulators use `checked_add` but only fail when the running sum exceeds `i64::MAX`. Bitcoin Knots rejects as soon as `nFees` leaves `MoneyRange`, which means the real boundary is `MAX_MONEY`, not `i64`. As written, `validate_block_with_context` and the chainstate path will treat `MAX_MONEY < total_fees_sats <= i64::MAX` as valid and then feed that larger value into the new coinbase reward-limit check. The new test at `packages/open-bitcoin-consensus/src/block/tests.rs:1462-1504` only exercises the later `i64` overflow boundary, so the intended parity rule is still untested.
**Fix:**
```rust
total_fees_sats = total_fees_sats
    .checked_add(fee.to_sats())
    .filter(|sum| (0..=MAX_MONEY).contains(sum))
    .ok_or_else(accumulated_fee_out_of_range)?;
```

Apply the same guard in both accumulators and add a regression that exceeds `MAX_MONEY` without overflowing `i64`, so the test locks in the same boundary that Knots enforces.
