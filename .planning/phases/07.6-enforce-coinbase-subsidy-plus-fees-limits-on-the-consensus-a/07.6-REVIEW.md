## Critical Issues

### CR-01: `check_tx_inputs` can wrap large input totals back into `MoneyRange`

**File:** `packages/open-bitcoin-consensus/src/context.rs:343-390`
**Issue:** `check_tx_inputs` updates `total_input_value` with plain `+=` before validating the new sum, and it computes `total_output_value` with unchecked `.sum()`. That makes the consensus result build-profile-dependent: in overflow-checking builds it can panic, and in release builds it wraps. The wrap is reachable with realistic input counts. For example, `8_785` `MAX_MONEY` inputs wrap to `1_755_926_290_448_384`, which still lies inside `0..=MAX_MONEY`, so the current guard can treat an invalid contextual transaction as having a valid input total.
**Fix:**
```rust
let value = input_context.spent_output.value.to_sats();
total_input_value = total_input_value
    .checked_add(value)
    .filter(|sum| (0..=MAX_MONEY).contains(sum))
    .ok_or_else(|| {
        tx_error(
            TxValidationResult::Consensus,
            "bad-txns-inputvalues-outofrange",
            None,
        )
    })?;

let total_output_value = transaction.outputs.iter().try_fold(0_i64, |sum, output| {
    sum.checked_add(output.value.to_sats())
}).ok_or_else(|| {
    tx_error(
        TxValidationResult::Consensus,
        "bad-txns-in-belowout",
        Some("output total out of range".to_string()),
    )
})?;
```

Add a regression in `packages/open-bitcoin-consensus/src/context/tests.rs` that constructs the smallest wrapping case instead of only the current non-wrapping `2 * MAX_MONEY` case.

### CR-02: `reorg` is not atomic when the replacement branch fails

**File:** `packages/open-bitcoin-chainstate/src/engine.rs:215-224`
**Issue:** `reorg` disconnects blocks from `self` and then connects replacement blocks directly into the same live chainstate. If any later `connect_block` call returns an error, the method exits without restoring the original tip, UTXO set, or undo map. A failed reorg attempt can therefore leave the node on a partially disconnected or partially replaced chain even though the operation reported failure.
**Fix:**
```rust
let snapshot = self.snapshot();
let mut staged = Self::from_snapshot(snapshot);

let mut transition = ChainTransition::default();
for block in disconnect_blocks {
    transition.disconnected.push(staged.disconnect_tip(block)?);
}
for anchored_block in replacement_branch {
    transition.connected.push(staged.connect_block(
        &anchored_block.block,
        anchored_block.chain_work,
        verify_flags,
        consensus_params,
    )?);
}

*self = staged;
Ok(transition)
```

Add a regression in `packages/open-bitcoin-chainstate/src/engine/tests.rs` that injects an invalid second replacement block and asserts `snapshot_before == snapshot_after`.

## Warnings

### WR-01: `disconnect_tip` mutates live state before its error paths are exhausted

**File:** `packages/open-bitcoin-chainstate/src/engine.rs:175-197`, `packages/open-bitcoin-chainstate/src/engine.rs:442-465`
**Issue:** `disconnect_tip` removes the undo entry before validating its shape, then mutates `self.utxos` in place via `remove_transaction_outputs` and `restore_non_coinbase_inputs`. If any of those checks fail, the method returns with `undo_by_block` already changed and the UTXO set potentially partially updated. The current tests cover these error branches, but they only assert the error variant and do not verify snapshot preservation.
**Fix:**
```rust
let maybe_block_undo = self.undo_by_block.get(&tip.block_hash).cloned().ok_or(...)?;
let mut next_utxos = self.utxos.clone();

// Validate and apply the disconnect into next_utxos first.

self.utxos = next_utxos;
self.undo_by_block.remove(&tip.block_hash);
self.active_chain.pop();
```

At minimum, move destructive `remove` calls until after validation succeeds. Prefer staging the whole disconnect against cloned state so the method matches the atomic behavior already used by `connect_block_with_current_time`.
