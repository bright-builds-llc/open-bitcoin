## Summary

Knots already enforces the contract this phase needs: `ConnectBlock` accumulates non-coinbase fees, checks `MoneyRange(nFees)`, computes `GetBlockSubsidy(height) + nFees`, and rejects an overpaying coinbase with `bad-cb-amount`; Open Bitcoin currently computes per-transaction fees but drops them in both `validate_block_with_context()` and `connect_block_with_current_time()`. [CITED: packages/bitcoin-knots/src/validation.cpp; CITED: packages/bitcoin-knots/src/validation.h; VERIFIED: packages/open-bitcoin-consensus/src/block.rs; VERIFIED: packages/open-bitcoin-consensus/src/transaction.rs; VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs; VERIFIED: .planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-VERIFICATION.md]

The clean implementation path is narrow. Keep a single pure reward-limit helper in the consensus crate, aggregate fees exactly once from the existing `validate_transaction_with_context()` return value, and teach the chainstate connect loop to accumulate those same fees before assigning `self.utxos`, `self.undo_by_block`, or `self.active_chain`. The current engine already uses cloned `next_utxos` and local undo state, so the planner does not need a wider transactional rewrite to satisfy the “no live mutation on failure” requirement. [VERIFIED: packages/open-bitcoin-consensus/src/transaction.rs; VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs; VERIFIED: .planning/phases/07.6-enforce-coinbase-subsidy-plus-fees-limits-on-the-consensus-a/07.6-CONTEXT.md]

The non-obvious planning detail is subsidy configuration. Open Bitcoin has no subsidy or halving helper and no halving-interval field in `ConsensusParams`, while Knots drives `GetBlockSubsidy()` from consensus params and the phase context explicitly allows tiny-halving fixtures for deterministic tests. The planner should therefore include a typed `subsidy_halving_interval` addition rather than hard-coding `210_000` in multiple places. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; CITED: packages/bitcoin-knots/src/validation.cpp; VERIFIED: .planning/phases/07.6-enforce-coinbase-subsidy-plus-fees-limits-on-the-consensus-a/07.6-CONTEXT.md]

**Primary recommendation:** Add `subsidy_halving_interval` to `ConsensusParams`, implement one Knots-style reward-limit helper in `block.rs`, accumulate fees from the existing transaction-validation return path on both consensus and chainstate surfaces, and prove the fix with exact-fee, `+1` overpay, and unchanged-snapshot tests plus `bash scripts/verify.sh`. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; VERIFIED: packages/open-bitcoin-consensus/src/transaction.rs; VERIFIED: packages/open-bitcoin-consensus/src/block.rs; VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs; CITED: packages/bitcoin-knots/src/validation.cpp]

### Pattern 1: Shared Reward-Limit Helper
**What:** Keep one pure helper in the consensus crate that computes block subsidy, compares coinbase outputs against `subsidy(height) + total_fees`, and returns a `BlockValidationError` with `bad-cb-amount` on failure. [CITED: packages/bitcoin-knots/src/validation.cpp; VERIFIED: packages/open-bitcoin-consensus/src/block.rs; VERIFIED: packages/open-bitcoin-consensus/src/validation.rs]
**When to use:** Use it from `validate_block_with_context()` and from the chainstate connect path after fee accumulation is complete. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs]
**Example:**
```rust
// Source pattern: packages/bitcoin-knots/src/validation.cpp
// and packages/open-bitcoin-consensus/src/block.rs
pub fn enforce_coinbase_reward_limit(
    block: &Block,
    height: u32,
    total_fees_sats: i64,
    params: &ConsensusParams,
) -> Result<(), BlockValidationError> {
    let subsidy_sats = block_subsidy(height, params).to_sats();
    let limit_sats = subsidy_sats
        .checked_add(total_fees_sats)
        .expect("subsidy + fees must stay within i64");
    let coinbase_value_sats = block.transactions[0]
        .outputs
        .iter()
        .map(|output| output.value.to_sats())
        .sum::<i64>();

    if coinbase_value_sats > limit_sats {
        return Err(block_error(
            BlockValidationResult::Consensus,
            "bad-cb-amount",
            Some(format!(
                "coinbase pays too much (actual={coinbase_value_sats} vs limit={limit_sats})"
            )),
        ));
    }

    Ok(())
}
```

- **Second fee calculator:** Do not recompute block fees from outputs or a second prevout traversal when `validate_transaction_with_context()` already returns the fee. [VERIFIED: packages/open-bitcoin-consensus/src/transaction.rs; VERIFIED: .planning/phases/07.6-enforce-coinbase-subsidy-plus-fees-limits-on-the-consensus-a/07.6-CONTEXT.md]

### Pitfall 1: Missing Subsidy Configuration
**What goes wrong:** The implementation hard-codes `50 * COIN` forever or hard-codes `210_000` in one helper and one test. [CITED: packages/bitcoin-knots/src/validation.cpp; VERIFIED: packages/open-bitcoin-consensus/src/context.rs]
**Why it happens:** Open Bitcoin currently has no subsidy helper and no halving-interval field in `ConsensusParams`. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; VERIFIED: rg -n "subsidy|halving" packages/open-bitcoin-consensus packages/open-bitcoin-chainstate packages/open-bitcoin-primitives packages/open-bitcoin-codec]
**How to avoid:** Add `subsidy_halving_interval` to `ConsensusParams`, default it to the Knots schedule, and cover the `halvings >= 64 => 0` branch directly. [CITED: packages/bitcoin-knots/src/validation.cpp; VERIFIED: packages/open-bitcoin-consensus/src/context/tests.rs]
**Warning signs:** `rg subsidy` still returns no open-bitcoin matches after the phase, or the new tests need unrealistic heights because the schedule is not configurable. [VERIFIED: rg -n "subsidy|halving" packages/open-bitcoin-consensus packages/open-bitcoin-chainstate packages/open-bitcoin-primitives packages/open-bitcoin-codec; VERIFIED: .planning/phases/07.6-enforce-coinbase-subsidy-plus-fees-limits-on-the-consensus-a/07.6-CONTEXT.md]

### Pitfall 2: Using `Amount` for the Reward Limit
**What goes wrong:** A valid comparison path is turned into an artificial overflow because `Amount::from_sats(subsidy + fees)` rejects values above `MAX_MONEY`. [VERIFIED: packages/open-bitcoin-primitives/src/amount.rs]
**Why it happens:** `Amount` encodes Bitcoin money-range invariants, but Knots compares coinbase value against a raw `CAmount` sum of subsidy plus fees. [VERIFIED: packages/open-bitcoin-primitives/src/amount.rs; CITED: packages/bitcoin-knots/src/validation.cpp]
**How to avoid:** Keep per-transaction fees and subsidy as typed amounts, but compare raw satoshis with `checked_add` for the block-level limit. [VERIFIED: packages/open-bitcoin-primitives/src/amount.rs; VERIFIED: packages/open-bitcoin-consensus/src/transaction.rs]
**Warning signs:** The helper calls `Amount::from_sats(subsidy_sats + total_fees_sats)` or uses saturating arithmetic. [VERIFIED: packages/open-bitcoin-primitives/src/amount.rs]

### Pitfall 3: Fixing the Test Helper but Not the Live Path
**What goes wrong:** `validate_block_with_context()` rejects an overpaying coinbase in tests, but `connect_block_with_current_time()` still accepts it into the staged chainstate. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs]
**Why it happens:** The engine does not call `validate_block_with_context()`; it only calls `check_block_contextual()` and then validates transactions one by one. [VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs; VERIFIED: packages/open-bitcoin-consensus/src/block.rs]
**How to avoid:** Reuse the same shared consensus helper from `engine.rs` after fee accumulation and before final assignment. [VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs; VERIFIED: .planning/phases/07.6-enforce-coinbase-subsidy-plus-fees-limits-on-the-consensus-a/07.6-CONTEXT.md]
**Warning signs:** Only consensus tests are added, or the chainstate test does not compare the post-failure snapshot to the pre-failure snapshot. [VERIFIED: packages/open-bitcoin-chainstate/src/engine.rs; VERIFIED: packages/open-bitcoin-chainstate/src/engine/tests.rs]

### Pitfall 4: Reject-Reason Drift
**What goes wrong:** The fix rejects correctly but emits repo-invented strings instead of Knots-compatible ones. [CITED: packages/bitcoin-knots/src/validation.cpp]
**Why it happens:** Open Bitcoin’s typed error surface makes it easy to add nearby strings without consulting the baseline. [VERIFIED: packages/open-bitcoin-consensus/src/validation.rs; VERIFIED: packages/open-bitcoin-consensus/src/block.rs]
**How to avoid:** Keep `bad-cb-amount` for overpaying coinbases, and prefer Knots’ `bad-txns-accumulated-fee-outofrange` wording for fee-accumulator overflow when feasible. [CITED: packages/bitcoin-knots/src/validation.cpp]
**Warning signs:** New reject reasons do not appear anywhere in `packages/bitcoin-knots/src/validation.cpp`. [CITED: packages/bitcoin-knots/src/validation.cpp]

fn block_subsidy(height: u32, params: &ConsensusParams) -> Amount {
    let halvings = height / params.subsidy_halving_interval;
