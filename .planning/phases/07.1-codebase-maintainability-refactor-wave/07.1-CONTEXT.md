<domain>
## Phase Boundary

Phase 07.1 is a maintainability-only insertion between the completed wallet
phase and the upcoming RPC/CLI/config phase. It does not add new operator
surfaces, parity claims, or feature scope. It only:

1. extracts inline unit suites from oversized first-party Rust source files,
2. splits `packages/open-bitcoin-wallet/src/wallet.rs` into coherent submodules,
3. splits `packages/open-bitcoin-consensus/src/script.rs` into coherent
   submodules.

Broader cleanup work such as early-return sweeps, magic-number cleanup, panic
or illegal-state sweeps, or new RPC/CLI behavior stays out of scope.

</domain>

<decisions>
## Implementation Decisions

### Phase sequencing and scope
- **D-01:** Execute the phase in three sequential slices: tests-only
  extraction first, then the wallet production refactor, then the consensus
  script production refactor.
- **D-02:** Treat the tests-only wave as structural only: move inline unit
  suites into sibling Rust test modules and avoid production refactors in that
  wave unless a minimal module declaration is required to keep tests compiling.
- **D-03:** If moving tests out is enough to bring a production file below the
  oversized-file trigger and the file remains cohesive, stop there in this
  phase rather than forcing extra production splits.

### Wallet module boundary
- **D-04:** Preserve the public wallet crate surface exposed through
  `packages/open-bitcoin-wallet/src/lib.rs`; the refactor may move
  implementation, but it should not rename or remove the wallet-facing types
  and entrypoints.
- **D-05:** Keep `wallet.rs` as the module entry file and move implementation
  behind child files organized by responsibility, with scan, build, and sign as
  the preferred first split boundaries.
- **D-06:** Carry forward the Phase 7 wallet boundary: persistence stays
  adapter-owned and signing continues to reuse the consensus sighash and
  verifier surface rather than inventing a second wallet-specific engine.

### Consensus script module boundary
- **D-07:** Preserve the public consensus script surface in `script.rs`;
  `ScriptError`, `ScriptInputVerificationContext`, `eval_script`,
  `verify_script`, `verify_input_script`, and the sigop entrypoints must remain
  available from the same top-level module.
- **D-08:** Keep `verify_input_script` as the single canonical spending-path
  verifier and split the implementation under it by responsibility rather than
  by adding parallel verifier entrypoints.
- **D-09:** The preferred first split boundaries for `script.rs` are low-level
  helpers, legacy execution, witness handling, taproot or tapscript handling,
  and sigop counting.

### Verification and workflow
- **D-10:** Use crate-targeted tests and clippy only as iteration aids; the
  phase is not complete until `bash scripts/verify.sh` passes.
- **D-11:** Because existing 07.1 plans were created before formal discuss
  provenance existed, regenerate the plans after this context capture instead of
  executing the current direct-fallback plans.

### the agent's Discretion
- Exact child-file names within the wallet and script module trees stay at the
  agent's discretion as long as they follow the repo's `foo.rs` plus `foo/`
  pattern, keep public APIs stable, and improve cohesion instead of scattering
  helpers arbitrarily.

### Folded Todos
- **Refactor oversized files under limits** — This todo is the direct maintainability
  driver for Phase 07.1. Its problem statement about oversized first-party Rust
  files becomes explicit scope for the tests-only extraction wave plus the
  dedicated `wallet.rs` and `script.rs` refactors.

### Established Patterns
- Pure-core crates stay free of direct I/O and runtime effects.
- Multi-file Rust modules should use `foo.rs` plus `foo/` rather than
  `foo/mod.rs`.
- The canonical verifier stays centralized in `verify_input_script`; segwit and
  taproot extend that path rather than branching around it.

### Integration Points
- `packages/open-bitcoin-wallet/src/lib.rs` must continue to re-export the same
  wallet API after the internal split.
- `packages/open-bitcoin-consensus/src/lib.rs` must continue to re-export the
  same script-verification API after the internal split.
- `bash scripts/verify.sh` is the final gate that proves the refactor did not
  break workspace formatting, lint, build, tests, Bazel smoke builds, or
  coverage.
