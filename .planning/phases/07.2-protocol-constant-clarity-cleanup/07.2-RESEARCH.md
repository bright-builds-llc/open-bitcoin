# Phase 07.2: Protocol Constant Clarity Cleanup - Research

**Researched:** 2026-04-18  
**Domain:** targeted protocol constant cleanup in consensus, primitives, and
networking crates  
**Confidence:** HIGH

## Summary

- The phase does not need a broad discovery pass. The in-scope residual
  hotspots are already concrete and bounded: the `OP_SUCCESS` allowlist in
  `script/opcodes.rs`, default consensus parameter literals in `context.rs`,
  and repeated serialized-size literals in `context.rs`, `sighash.rs`, and
  `message.rs`.
- The best abstraction boundary for the repeated `36`-byte literal is the
  owning wire type, not a free-floating module constant. `OutPoint` is the
  right home for prevout size and `InventoryVector` is the right home for
  inventory-entry size.
- The existing code already provides one of the needed semantic anchors:
  `TransactionInput::SEQUENCE_LOCKTIME_GRANULARITY` in
  `packages/open-bitcoin-primitives/src/transaction.rs`. Reusing it avoids
  duplicating the sequence-locktime granularity value in `ConsensusParams`.
- The `OP_SUCCESS` cleanup should remain behavior-neutral. The safest rewrite is
  to express the predicate in opcode-domain terms with named singleton
  constants plus explicit hex ranges, then add boundary assertions in
  `script/tests.rs` so the extracted names cannot silently widen or narrow the
  allowlist.
- `bash scripts/verify.sh` remains the final truth gate for this phase because
  it exercises format, clippy, build, tests, Bazel smoke checks, coverage, and
  architecture-policy enforcement across the touched crates.
