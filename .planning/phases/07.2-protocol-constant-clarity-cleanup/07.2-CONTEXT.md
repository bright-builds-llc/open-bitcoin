<domain>
## Phase Boundary

Phase 07.2 is a narrow maintainability insertion between the completed Phase
07.1 refactor wave and the upcoming Phase 8 interface work. It does not add
new operator surfaces, new consensus or networking behavior, or a repo-wide
style sweep. It only:

1. replaces the audited protocol-significant magic numbers that still obscure
   intent in non-test code,
2. replaces duplicated serialized-size literals where the size belongs to a
   concrete domain type,
3. preserves existing behavior and tightens auditability before CLI or RPC work
   builds on these crates.

</domain>

<decisions>
## Implementation Decisions

### Scope and targeting
- **D-01:** Limit the phase to the concrete residual hotspots surfaced by the
  audit: `packages/open-bitcoin-consensus/src/script/opcodes.rs`,
  `packages/open-bitcoin-consensus/src/context.rs`,
  `packages/open-bitcoin-consensus/src/sighash.rs`, and
  `packages/open-bitcoin-network/src/message.rs`, plus the domain-type homes
  needed for shared constants.
- **D-02:** Keep obviously self-explanatory literals inline when extraction
  would reduce readability, including fixed-width hash arrays and
  `read_array::<32>()`-style calls that already encode their meaning at the
  call site.

### Constant placement
- **D-03:** Prefer type-owned or domain-owned constants over unrelated globals.
  Serialized outpoint size belongs with `OutPoint`, serialized inventory-vector
  size belongs with `InventoryVector`, consensus default parameters belong with
  `ConsensusParams`, and opcode-domain names belong in `script/opcodes.rs`.
- **D-04:** Reuse existing named constants instead of minting duplicates,
  especially `TransactionInput::SEQUENCE_LOCKTIME_GRANULARITY`.

### Behavior and verification
- **D-05:** Treat this as a clarity-only pass. The phase must not change
  accepted or rejected consensus behavior, network payload layout, or public
  crate surfaces.
- **D-06:** Add or adjust tests only where they materially guard extracted
  constant boundaries, especially the Taproot `OP_SUCCESS` allowlist. Do not
  add cosmetic tests that only restate constant values.

### the agent's Discretion
- Exact constant names remain at the agent's discretion as long as they encode
  protocol meaning directly and remain local to the owning type or module.

### Folded Todos
- **Sweep magic numbers for clarity** — This todo becomes the direct driver for
  Phase 07.2, but the phase scope is intentionally narrowed to the audited
  protocol-significant leftovers instead of a repo-wide literal sweep.

</decisions>

<specifics>
## Specific Ideas

- The audit found three concrete residual gaps: bare decimal `OP_SUCCESS`
  values and ranges in `script/opcodes.rs`, unnamed consensus default literals
  in `context.rs`, and duplicated `36`-byte serialized-size literals in
  `context.rs`, `sighash.rs`, and `message.rs`.
- Shared-size constants should be introduced at the owning type rather than in
  consensus or networking helpers, so the executor should prefer
  `OutPoint::SERIALIZED_LEN` and `InventoryVector::SERIALIZED_LEN`.
- The `OP_SUCCESS` rewrite should stay in opcode-domain terms and add boundary
  assertions so the extracted names do not accidentally change the BIP342
  allowlist.

</specifics>
