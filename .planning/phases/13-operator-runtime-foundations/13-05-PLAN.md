<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JSONC file text -> typed Open Bitcoin config | User-editable JSONC is untrusted input and must parse into typed structs. |
| environment variables -> config precedence | Environment can override JSONC and baseline config in non-interactive runs. |
| bitcoin.conf -> baseline-compatible runtime config | Existing Bitcoin/Knots config files must not be polluted with Open Bitcoin-only application settings. |
| cookie files -> auth fallback | Cookie files provide credentials and must remain an auth source, not a general settings source. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-13-05-01 | T | `parse_open_bitcoin_jsonc_config` | mitigate | Parse JSONC through `jsonc-parser` into serde structs with `deny_unknown_fields`, and test comments/trailing commas plus unknown-field rejection. |
| T-13-05-02 | T | `ConfigPrecedence` | mitigate | Encode and test the exact source order `CLI flags > environment > Open Bitcoin JSONC > bitcoin.conf > cookies > defaults`. |
| T-13-05-03 | I | config/status surfaces | mitigate | Keep RPC password and cookie contents out of Open Bitcoin JSONC contracts; cookie files remain credential sources only. |
| T-13-05-04 | R | bitcoin.conf compatibility | mitigate | Add tests proving Open Bitcoin-only keys such as `dashboard` or `service` are rejected by the baseline-compatible `bitcoin.conf` loader. |
</threat_model>
