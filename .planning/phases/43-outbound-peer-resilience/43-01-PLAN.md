# Plan 43-01: Backoff Visibility And Mixed Peer Failure Resilience

## Objective

Make outbound sync peer resilience truthful in deterministic status and telemetry
by reporting configured peer targets separately from observed peers, surfacing
retry-backoff waits as peer outcomes, and proving mixed peer failures rotate to
replacement peers without corrupting durable state or exiting the sync runtime.

## Requirements

- PEER-02: Daemon sync enforces bounded outbound peer counts and rotates
  unhealthy peers with stable backoff, stall, and retry reasons.
- PEER-04: Daemon sync handles mixed peer failures, disconnects, timeouts,
  invalid data, and peer replacement without corrupting durable state or exiting
  unexpectedly.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public peer input -> sync runtime | Untrusted peers can fail connects, stall, disconnect, or send invalid data. |
| Retry/backoff state -> operator telemetry | Runtime health decisions become status, logs, metrics, and reports. |
| Peer replacement -> durable state | The runtime continues after unhealthy peers while preserving validated chainstate. |
| Configured peer limits -> runtime actions | Operator-configured outbound target values bound useful peer connection slots. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-43-01-01 | Spoofing | Peer telemetry | mitigate | Project waiting/backoff states from runtime-owned `PeerSyncOutcome` enums instead of free-form peer text. |
| T-43-01-02 | Tampering | Durable sync state | mitigate | Mixed-failure handling records outcomes but only validated headers/blocks can advance durable progress. |
| T-43-01-03 | Repudiation | Failure diagnosis | mitigate | Persist stable failure reasons, health signals, and structured logs for stall, retry, backoff, and invalid data. |
| T-43-01-04 | Denial of Service | Outbound peer loop | mitigate | Preserve `target_outbound_peers` stopping behavior for successful slots and keep backoff waits from consuming connection attempts. |
| T-43-01-05 | Information Disclosure | Logs/status | mitigate | Waiting/backoff messages include bounded retry metadata and avoid private datadir, RPC credential, or payload data. |
</threat_model>

## Success Criteria

- Direct and durable sync status can show observed outbound peers below the
  configured target without losing the configured bound.
- Backoff-skipped peers are visible as waiting with `retry_backoff`.
- Stalled/failed peers rotate to replacement peers when candidates are
  available.
- Mixed connection failure, invalid data, and replacement success produce a
  summary and coherent durable state.
