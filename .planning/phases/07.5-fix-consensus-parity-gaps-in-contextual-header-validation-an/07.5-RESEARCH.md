  `time-too-old`, and `time-too-new` outcomes on the active contextual block

  inputs needed for parity, extend it with the smallest explicit context
  plumbing required. Prefer adding the missing contextual data to typed
  validation context structures over reading clocks or chain globals directly.

- **D-05:** On non-strict legacy verification paths, parse ECDSA signatures with
  lax DER semantics that match Knots instead of silently treating all non-DER
  encodings as false after strict parsing fails.
- **D-06:** Preserve the existing strict behavior gates: `STRICTENC`, `DERSIG`,
  and `LOW_S` must still enforce strict DER and low-S behavior exactly where the
  flags require it, and hybrid public-key normalization behavior must remain
  intact.

  `bash scripts/verify.sh` passes, and a follow-up review/verifier pass confirms

## Summary

Phase 07.5 is a narrow parity repair, not a new consensus feature slice. The two gaps are real on the active path today: `check_block_header_contextual()` only enforces `time-too-old`, and `parse_ecdsa_signature_for_verification()` currently treats non-strict DER parse failures as plain false instead of accepting Knots-compatible lax DER where appropriate. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; packages/open-bitcoin-consensus/src/signature.rs; .planning/phases/07.4-sweep-the-codebase-for-let-else-opportunities/07.4-REVIEW.md]

Planning needs to account for one structural fact on the header side: the current first-party Rust codebase has no `GetNextWorkRequired()` equivalent, and `ConsensusParams` carries no proof-of-work retarget parameters or current-time input today. That means the planner should allocate explicit work for minimal context or parameter expansion, not just a one-line guard insertion. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; packages/open-bitcoin-consensus/src/block.rs; packages/open-bitcoin-chainstate/src/engine.rs; packages/bitcoin-knots/src/pow.cpp]

Planning is simpler on the signature side than the review suggested. The existing `secp256k1` dependency already resolves to `0.31.1`, and that API already exposes `ecdsa::Signature::from_der_lax()`, which wraps lax DER parsing specifically for historical Bitcoin validation. The planner should therefore prefer reusing the current crate and preserve the existing strict gate split instead of scheduling a custom parser. [VERIFIED: packages/open-bitcoin-consensus/Cargo.toml; packages/Cargo.lock; /Users/peterryszkiewicz/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/secp256k1-0.31.1/src/ecdsa/mod.rs]

**Primary recommendation:** Add minimal typed context for `current_time` plus the proof-of-work schedule inputs needed by a consensus-local `next_work_required` helper, implement Knots-compatible `bad-diffbits` and `time-too-new` checks in `check_block_header_contextual()`, and switch non-strict legacy ECDSA parsing to `ecdsa::Signature::from_der_lax()` while keeping `STRICTENC`, `DERSIG`, `LOW_S`, and existing public-key behavior unchanged. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; packages/open-bitcoin-consensus/src/block.rs; packages/open-bitcoin-consensus/src/signature.rs; packages/bitcoin-knots/src/validation.cpp; packages/bitcoin-knots/src/pow.cpp; packages/bitcoin-knots/src/pubkey.cpp; packages/bitcoin-knots/src/script/interpreter.cpp; /Users/peterryszkiewicz/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/secp256k1-0.31.1/src/ecdsa/mod.rs]

### Pattern 1: Typed Context Injection For Contextual Header Rules

**What:** Extend the typed block-validation context so the consensus core receives all data needed for `bad-diffbits`, `time-too-old`, and `time-too-new` without reading runtime state directly. [VERIFIED: .planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-CONTEXT.md; packages/open-bitcoin-consensus/src/context.rs; packages/bitcoin-knots/src/validation.cpp; packages/bitcoin-knots/src/pow.cpp]

**When to use:** Use this pattern for the header fix itself and for any future contextual consensus check that depends on previous-chain state or a caller-provided clock. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; .planning/phases/03-consensus-validation-engine/03-03-SUMMARY.md]

**Example:**
```rust
// Source: packages/bitcoin-knots/src/validation.cpp + packages/bitcoin-knots/src/pow.cpp
pub struct BlockValidationContext {
    pub height: u32,
    pub previous_header: BlockHeader,
    pub previous_median_time_past: i64,
    pub current_time: i64,
    pub consensus_params: ConsensusParams,
}

pub fn check_block_header_contextual(
    header: &BlockHeader,
    context: &BlockValidationContext,
) -> Result<(), BlockValidationError> {
    let expected_bits = next_work_required(&context.previous_header, header, &context.consensus_params)?;
    if header.bits != expected_bits {
        return Err(block_error(
            BlockValidationResult::InvalidHeader,
            "bad-diffbits",
            Some("incorrect proof of work".to_string()),
        ));
    }

    if i64::from(header.time) <= context.previous_median_time_past {
        return Err(block_error(
            BlockValidationResult::InvalidHeader,
            "time-too-old",
            Some("block's timestamp is too early".to_string()),
        ));
    }

    if i64::from(header.time) > context.current_time + MAX_FUTURE_BLOCK_TIME {
        return Err(block_error(
            BlockValidationResult::TimeFuture,
            "time-too-new",
            Some("block timestamp too far in the future".to_string()),
        ));
    }

    Ok(())
}
```

### Pattern 2: One Strict-Or-Lax Parse Gate For Legacy ECDSA

**What:** Keep one helper responsible for selecting strict DER parsing under `STRICTENC` or `DERSIG` or `LOW_S`, and use lax DER parsing only on the non-strict legacy path. [VERIFIED: packages/open-bitcoin-consensus/src/signature.rs; packages/bitcoin-knots/src/pubkey.cpp; packages/bitcoin-knots/src/script/interpreter.cpp; /Users/peterryszkiewicz/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/secp256k1-0.31.1/src/ecdsa/mod.rs]

**When to use:** Use this only inside `parse_ecdsa_signature_for_verification()` so all legacy script entrypoints inherit the same parity behavior. [VERIFIED: packages/open-bitcoin-consensus/src/signature.rs]

**Example:**
```rust
// Source: packages/bitcoin-knots/src/pubkey.cpp + packages/bitcoin-knots/src/script/interpreter.cpp
fn parse_ecdsa_signature_for_verification(
    bytes: &[u8],
    flags: ScriptVerifyFlags,
) -> Result<Option<ParsedEcdsaSignature>, SignatureError> {
    let Some((&hash_type, der)) = bytes.split_last() else {
        return Ok(None);
    };

    let sighash_type = SigHashType::from_u32(u32::from(hash_type));
    if flags.contains(ScriptVerifyFlags::STRICTENC) {
        validate_legacy_sighash_type(sighash_type)?;
    }

    let require_strict_der = flags.contains(ScriptVerifyFlags::DERSIG)
        || flags.contains(ScriptVerifyFlags::LOW_S)
        || flags.contains(ScriptVerifyFlags::STRICTENC);

    let maybe_signature = if require_strict_der {
        Some(ecdsa::Signature::from_der(der).map_err(|_| SignatureError::InvalidDer)?)
    } else {
        ecdsa::Signature::from_der_lax(der).ok()
    };

    let Some(signature) = maybe_signature else {
        return Ok(None);
    };

    if flags.contains(ScriptVerifyFlags::LOW_S) {
        let mut normalized = signature;
        normalized.normalize_s();
        if normalized != signature {
            return Err(SignatureError::NonLowS);
        }
    }

    Ok(Some(ParsedEcdsaSignature { signature, sighash_type }))
}
```

### Anti-Patterns to Avoid

- **Clock reads inside consensus:** Do not call wall-clock APIs from `open-bitcoin-consensus`; inject time through `BlockValidationContext` instead. [VERIFIED: .planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-CONTEXT.md; ../coding-and-architecture-requirements/standards/core/architecture.md; .planning/PROJECT.md]
- **Rechecking PoW without next-work comparison:** `check_proof_of_work()` only validates the hash against the header’s own target; it does not prove that `header.bits` equals Knots’ required next target. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; packages/bitcoin-knots/src/pow.cpp; packages/bitcoin-knots/src/validation.cpp]
- **Hand-rolled lax DER parsing:** The existing `secp256k1` crate already exposes `from_der_lax()`, so a custom parser adds risk without adding capability. [VERIFIED: /Users/peterryszkiewicz/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/secp256k1-0.31.1/src/ecdsa/mod.rs]
- **Broad `ConsensusParams` cleanup:** The phase context explicitly forbids widening scope; add only the PoW schedule fields the new helper actually needs. ## Common Pitfalls

### Pitfall 1: Treating `bad-diffbits` As A Duplicate PoW Check

**What goes wrong:** The implementation reuses `check_proof_of_work()` and concludes contextual difficulty is already enforced. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; packages/bitcoin-knots/src/validation.cpp]

**Why it happens:** `check_block_header()` already returns `bad-diffbits` for malformed compact targets, which can make it easy to miss that Knots also compares `block.nBits` to `GetNextWorkRequired()` in the contextual path. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; packages/bitcoin-knots/src/validation.cpp]

**How to avoid:** Plan a dedicated consensus helper for next-work calculation and use it from `check_block_header_contextual()`. [VERIFIED: packages/bitcoin-knots/src/pow.cpp; packages/open-bitcoin-consensus/src/block.rs]

**Warning signs:** A patch that adds `time-too-new` but still leaves `ConsensusParams` and `BlockValidationContext` unchanged is almost certainly incomplete. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; .planning/phases/07.4-sweep-the-codebase-for-let-else-opportunities/07.4-REVIEW.md]

### Pitfall 2: Forgetting That Current Rust Context Objects Lack PoW Inputs

**What goes wrong:** Planning assumes the current context already carries enough information for Knots-style next-work logic. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs]

**Why it happens:** `BlockValidationContext` already has `height`, `previous_header`, and `previous_median_time_past`, which looks close to complete until you compare it to Knots `GetNextWorkRequired()` and `time-too-new`. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; packages/bitcoin-knots/src/pow.cpp; packages/bitcoin-knots/src/validation.cpp]

**How to avoid:** Schedule an explicit task to extend typed context or parameters with `current_time` plus the proof-of-work schedule inputs the helper needs. [VERIFIED: packages/open-bitcoin-consensus/src/context.rs; packages/bitcoin-knots/src/pow.cpp]

**Warning signs:** The planned code changes touch only `block.rs` and not `context.rs` or `engine.rs`. [VERIFIED: packages/open-bitcoin-consensus/src/block.rs; packages/open-bitcoin-consensus/src/context.rs; packages/open-bitcoin-chainstate/src/engine.rs]

### Pitfall 3: Routing Strict Flags Through Lax DER Parsing

**What goes wrong:** A “simple” fix switches all legacy signature parsing to lax DER and silently weakens `DERSIG`, `LOW_S`, or `STRICTENC`. [VERIFIED: packages/bitcoin-knots/src/script/interpreter.cpp; packages/open-bitcoin-consensus/src/signature.rs]

**Why it happens:** Knots splits the behavior across `CheckSignatureEncoding()` and `CPubKey::Verify()`, so the strict-versus-non-strict contract lives across two files instead of one helper. [VERIFIED: packages/bitcoin-knots/src/script/interpreter.cpp; packages/bitcoin-knots/src/pubkey.cpp]

**How to avoid:** Keep the existing strict gate check first, then use lax parsing only on the non-strict legacy path. [VERIFIED: packages/bitcoin-knots/src/script/interpreter.cpp; packages/bitcoin-knots/src/pubkey.cpp; packages/open-bitcoin-consensus/src/signature.rs]

**Warning signs:** A patch no longer returns `SignatureError::InvalidDer` under `DERSIG` or `LOW_S`, or it changes the existing `LOW_S` normalization check. [VERIFIED: packages/open-bitcoin-consensus/src/signature.rs; packages/open-bitcoin-consensus/src/script/legacy.rs]

### Pitfall 4: Getting False-Versus-Error Mapping Wrong On Non-Strict Signatures

**What goes wrong:** The parser starts returning an error for all malformed non-strict signatures, which changes script-level behavior from “false” to a hard reject. [VERIFIED: packages/open-bitcoin-consensus/src/signature.rs; packages/open-bitcoin-consensus/src/script/legacy.rs]

**Why it happens:** The current helper returns `Ok(None)` for empty or unparsable non-strict signatures, while strict failures map to `SignatureError::InvalidDer`; both flows look similar in the helper but produce different script errors downstream. [VERIFIED: packages/open-bitcoin-consensus/src/signature.rs; packages/open-bitcoin-consensus/src/script/legacy.rs]

**How to avoid:** Preserve `Ok(None)` for non-strict parse failure, and add a regression that proves a known Knots-accepted lax DER signature becomes `Some(...)` under `ScriptVerifyFlags::NONE`. [VERIFIED: packages/open-bitcoin-consensus/src/signature.rs; packages/bitcoin-knots/src/test/data/script_tests.json; /Users/peterryszkiewicz/.cargo/registry/src/index.crates.io-1949cf8c6b5b557f/secp256k1-0.31.1/src/lib.rs]

**Warning signs:** Helper tests only assert strict errors and generic non-strict false cases, but never prove acceptance of a specific lax DER vector. [VERIFIED: packages/open-bitcoin-consensus/src/signature/tests.rs]

### Pitfall 5: Adding The Right Logic But No Phase-Specific Regression Evidence

**What goes wrong:** The code changes pass compile and broad verification, but nothing guards the exact parity gap that triggered the phase. [VERIFIED: .planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-CONTEXT.md]

**Why it happens:** `bash scripts/verify.sh` is broad and green, but it does not by itself prove that a specific contextual header or lax DER regression is now covered. [VERIFIED: scripts/verify.sh]

**How to avoid:** Plan one targeted contextual-header test and one targeted lax-DER regression vector, then rerun the follow-up review or verifier pass named in D-08. [VERIFIED: .planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-CONTEXT.md]

**Warning signs:** The plan mentions only `bash scripts/verify.sh` and no new named regression tests or parity vectors. [VERIFIED: .planning/phases/07.5-fix-consensus-parity-gaps-in-contextual-header-validation-an/07.5-CONTEXT.md]
