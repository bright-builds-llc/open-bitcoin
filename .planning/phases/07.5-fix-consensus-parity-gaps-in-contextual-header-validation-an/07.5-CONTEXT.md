  contextual header validation parity and non-strict lax DER verification
  parity.
- **D-02:** Allow minimal supporting-file changes only where the fixes require

  `packages/open-bitcoin-consensus/src/context.rs`,

  `packages/open-bitcoin-chainstate/src/engine.rs`, and targeted consensus test
  files, but do not widen into unrelated consensus cleanup, policy work, or
  Phase 8 interface behavior.

### Contextual header parity contract
- **D-03:** Treat Knots `ContextualCheckBlockHeader()` and
  `GetNextWorkRequired()` as the canonical behavior source for the header fix.
  The Open Bitcoin path must enforce the equivalent `bad-diffbits`,
  `time-too-old`, and `time-too-new` outcomes on the active contextual block
  validation path rather than only enforcing median-time-past.
- **D-04:** If the current Open Bitcoin contextual validation surface lacks the
  inputs needed for parity, extend it with the smallest explicit context
  plumbing required. Prefer adding the missing contextual data to typed
  validation context structures over reading clocks or chain globals directly.

### Legacy ECDSA parity contract
- **D-05:** On non-strict legacy verification paths, parse ECDSA signatures with
  lax DER semantics that match Knots instead of silently treating all non-DER
  encodings as false after strict parsing fails.
- **D-06:** Preserve the existing strict behavior gates: `STRICTENC`, `DERSIG`,
  and `LOW_S` must still enforce strict DER and low-S behavior exactly where the
  flags require it, and hybrid public-key normalization behavior must remain
  intact.

### Verification and completion
- **D-07:** Add targeted regression evidence for both fixes. The header change
  should be proven with contextual block-validation tests, and the signature
  change should be proven with a Knots-compatible lax DER regression vector or
  equivalent parity fixture on the non-strict path.
- **D-08:** The phase is not complete until the targeted consensus tests pass,
  `bash scripts/verify.sh` passes, and a follow-up review/verifier pass confirms
  the two parity findings are actually closed.

### the agent's Discretion
- Exact helper names, where to place any new parity regression vectors, and how
  to thread additional contextual validation inputs remain at the agent's
  discretion as long as the resulting surface stays typed, narrow, and
  behavior-neutral outside the two intended fixes.
- If one fix clearly needs a separate supporting helper or test module, the
  agent may add it, but only when it directly serves the two parity gaps.
