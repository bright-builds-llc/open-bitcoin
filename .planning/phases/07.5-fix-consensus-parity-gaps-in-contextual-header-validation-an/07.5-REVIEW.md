## Critical Issues

### CR-01: Min-Difficulty Recovery Cannot Match Knots After a Special Block

**Files:** `packages/open-bitcoin-consensus/src/block/difficulty.rs:31-43`, `packages/open-bitcoin-consensus/src/context.rs:138-150`, `packages/open-bitcoin-chainstate/src/engine.rs:94-100`, `packages/open-bitcoin-chainstate/src/engine.rs:231-250`

**Issue:** On networks with `allow_min_difficulty_blocks = true` and retargeting enabled, Knots does not reuse the immediately previous `nBits` after an on-time block follows a special min-difficulty block. It walks back to the last non-special block and reuses that value instead. This implementation always returns `context.previous_header.bits`, and the current `BlockValidationContext` never carries the ancestor data needed to recover the last non-special target. Result: once a late block drops to `pow_limit_bits`, the next on-time block is validated against the wrong difficulty and can fork from the Knots baseline.

**Fix:**
```rust
pub struct BlockValidationContext {
    pub height: u32,
    pub previous_header: BlockHeader,
    pub maybe_retarget_anchor: Option<RetargetAnchor>,
    pub maybe_last_non_min_difficulty_bits: Option<u32>,
    pub previous_median_time_past: i64,
    pub current_time: i64,
    pub consensus_params: ConsensusParams,
}

fn next_work_required(
    header: &BlockHeader,
    context: &BlockValidationContext,
) -> Result<u32, BlockValidationError> {
    if i64::from(context.height) % interval != 0 {
        if allow_min_difficulty_block {
            return Ok(consensus_params.pow_limit_bits);
        }

        if let Some(bits) = context.maybe_last_non_min_difficulty_bits {
            return Ok(bits);
        }

        return Ok(context.previous_header.bits);
    }
    // ...
}
```

Populate `maybe_last_non_min_difficulty_bits` in chainstate by walking backward until the interval boundary or the first header whose bits are not `pow_limit_bits`, then add a regression test that covers the Knots `pow.cpp` "last non-special-min-difficulty-rules-block" path.

## Warnings

### WR-01: Default Block-Connection APIs Disable `time-too-new` Validation

**Files:** `packages/open-bitcoin-chainstate/src/engine.rs:55-68`, `packages/open-bitcoin-node/src/chainstate.rs:57-70`, `packages/open-bitcoin-node/src/network.rs:159-170`, `packages/open-bitcoin-consensus/src/block.rs:191-200`

**Issue:** The new convenience `connect_block` wrappers pass `current_time = block.header.time`. That makes the future-time guard in `check_block_header_contextual` unreachable for every call site that uses the default API, because `header.time > current_time + 7200` can never be true when both values come from the same header. Remote peer ingestion uses `connect_block_with_current_time`, but local block submission and reorg replacement still bypass the new consensus check.

**Fix:**
```rust
pub fn connect_block(
    &mut self,
    block: &Block,
    chain_work: u128,
    current_time: i64,
    verify_flags: ScriptVerifyFlags,
    consensus_params: ConsensusParams,
) -> Result<ChainPosition, ChainstateError> {
    self.connect_block_with_current_time(
        block,
        chain_work,
        current_time,
        verify_flags,
        consensus_params,
    )
}
```

Thread an actual clock or adjusted-time value through the local node entrypoints and remove the header-time default so all connection paths enforce the same future-time rule.

### WR-02: Incoming Wire Messages Ignore Network Magic

**File:** `packages/open-bitcoin-node/src/network.rs:130-145`

**Issue:** `receive_wire_message` decodes the full network header but then discards `parsed.header.magic`. A node configured for one network will accept payloads from another network as long as the checksum and command parse successfully. That breaks network isolation and lets the wrong transport feed headers, blocks, and transactions into the local peer state.

**Fix:**
```rust
let parsed = ParsedNetworkMessage::decode_wire(bytes)?;
if parsed.header.magic != self.local_config.magic {
    return Err(NetworkError::InvalidMagic {
        expected: self.local_config.magic,
        actual: parsed.header.magic,
    }
    .into());
}
```

Add a dedicated `NetworkError` variant for wrong-network magic and reject the packet before dispatching it to `receive_message`.

### WR-03: `PeerAction::Disconnect` Is Dropped on the Floor

**Files:** `packages/open-bitcoin-node/src/network.rs:213-223`, `packages/open-bitcoin-node/src/network.rs:227-272`

**Issue:** The wrapper converts peer-manager actions into outbound messages only. `collect_outbound` and `process_actions` ignore `PeerAction::Disconnect(_)`, so protocol violations that the peer manager explicitly wants disconnected, such as duplicate `version` messages, never reach the transport layer. The bad peer stays registered locally and the caller receives no signal that the connection should be torn down.

**Fix:**
```rust
pub enum ManagedPeerAction {
    Send(WireNetworkMessage),
    Disconnect { peer_id: PeerId, reason: DisconnectReason },
}

fn process_actions(
    &mut self,
    peer_id: PeerId,
    actions: Vec<PeerAction>,
    // ...
) -> Result<Vec<ManagedPeerAction>, ManagedNetworkError> {
    // map PeerAction::Disconnect into a surfaced action instead of dropping it
}
```

Either surface disconnect actions through the public API or handle them internally by removing the peer and returning a transport-level disconnect instruction.
