## Critical Issues

### CR-01: Block Connection Never Enforces Coinbase Subsidy Plus Fees

**File:** `packages/open-bitcoin-consensus/src/block.rs:261`, `packages/open-bitcoin-chainstate/src/engine.rs:114`, `packages/open-bitcoin-chainstate/src/engine.rs:286`, `packages/open-bitcoin-consensus/src/context.rs:323`
**Issue:** `check_tx_inputs()` and `validate_transaction_with_context()` compute a per-transaction fee, but the reviewed block-validation and block-connection paths discard it. `validate_block_with_context()` validates each non-coinbase transaction and returns `Ok(())` without summing fees, and `Chainstate::connect_block_with_current_time()` / `apply_non_coinbase_transaction()` add the coinbase outputs to the UTXO set without ever comparing the coinbase payout against `subsidy(height) + total_fees`. As written, a block whose coinbase over-claims value will pass contextual checks and be connected, which is a consensus break.
**Fix:**
```rust
let mut total_fees = Amount::from_sats(0).expect("zero");
for (transaction, transaction_context) in block.transactions.iter().skip(1).zip(transaction_contexts)
{
    let fee = validate_transaction_with_context(transaction, transaction_context)
        .map_err(|error| map_transaction_validation_error(transaction, error))?;
    total_fees = (total_fees + fee).map_err(|_| block_error(
        BlockValidationResult::Consensus,
        "bad-cb-amount",
        Some("total fees overflowed".to_string()),
    ))?;
}

let coinbase_value = block.transactions[0]
    .outputs
    .iter()
    .try_fold(Amount::from_sats(0).expect("zero"), |sum, output| sum + output.value)
    .map_err(|_| block_error(BlockValidationResult::Consensus, "bad-cb-amount", None))?;
let allowed = block_subsidy(block_context.height, &block_context.consensus_params) + total_fees;
if coinbase_value > allowed {
    return Err(block_error(
        BlockValidationResult::Consensus,
        "bad-cb-amount",
        Some("coinbase pays too much".to_string()),
    ));
}
```

## Warnings

### WR-01: Public `connect_block` Wrappers Disable `time-too-new` Rejection

**File:** `packages/open-bitcoin-chainstate/src/engine.rs:55`, `packages/open-bitcoin-node/src/chainstate.rs:57`, `packages/open-bitcoin-node/src/network.rs:159`
**Issue:** The convenience `connect_block()` wrappers pass `i64::from(block.header.time)` as the `current_time` argument. That makes `check_block_header_contextual()`'s future-time guard unreachable for those APIs, because `header.time > current_time + 7200` can never be true when `current_time == header.time`. Remote peer blocks go through `connect_block_with_current_time(..., timestamp, ...)`, so only locally connected blocks bypass the future-time policy. Any caller using `ManagedPeerNetwork::connect_local_block()` or `ManagedChainstate::connect_block()` can therefore admit blocks that should be rejected as `time-too-new`.
**Fix:**
```rust
pub fn connect_block_with_clock(
    &mut self,
    block: &Block,
    chain_work: u128,
    now: i64,
    verify_flags: ScriptVerifyFlags,
    consensus_params: ConsensusParams,
) -> Result<ChainPosition, ChainstateError> {
    self.connect_block_with_current_time(
        block,
        chain_work,
        now,
        verify_flags,
        consensus_params,
    )
}
```

Use a real wall-clock value at the adapter boundary, or remove the zero-argument wrappers so callers cannot accidentally opt out of the future-time check.

### WR-02: `disconnect_tip` / `reorg` Leave Chainstate Partially Mutated on Error

**File:** `packages/open-bitcoin-chainstate/src/engine.rs:145`, `packages/open-bitcoin-chainstate/src/engine.rs:183`, `packages/open-bitcoin-node/src/chainstate.rs:103`
**Issue:** `disconnect_tip()` removes undo data and starts mutating `self.utxos` before all failure conditions are known. If any later check fails, the method returns `Err` after partially modifying live state. `reorg()` compounds this by disconnecting blocks on the live chainstate and then calling `connect_block()` one replacement block at a time; if a replacement block fails, the node is left on a partially rewound or partially reconnected branch. `ManagedChainstate::reorg()` only persists after success, so an error can also leave in-memory state diverged from the stored snapshot.
**Fix:**
```rust
pub fn reorg(...) -> Result<ChainTransition, ChainstateError> {
    let mut working = self.clone();
    let mut transition = ChainTransition::default();

    for block in disconnect_blocks {
        transition.disconnected.push(working.disconnect_tip(block)?);
    }
    for anchored_block in replacement_branch {
        transition.connected.push(working.connect_block_with_current_time(
            &anchored_block.block,
            anchored_block.chain_work,
            current_time,
            verify_flags,
            consensus_params,
        )?);
    }

    *self = working;
    Ok(transition)
}
```

Apply the same transactional pattern to `disconnect_tip()` or validate the undo shape before mutating maps so error paths cannot corrupt the in-memory chainstate.
