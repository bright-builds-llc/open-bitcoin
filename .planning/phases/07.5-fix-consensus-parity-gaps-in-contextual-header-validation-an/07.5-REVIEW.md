## Critical Issues

### CR-01: Retarget Heights Never Compute A New Difficulty

**File:** `packages/open-bitcoin-consensus/src/block.rs:304-329`, `packages/open-bitcoin-consensus/src/block/tests.rs:880-909`  
**Issue:** `next_work_required()` always returns `context.previous_header.bits` at retarget boundaries, even when `no_pow_retargeting` is `false`. That means every difficulty transition block on a retargeting network is validated against the previous target instead of the retargeted one, which is a consensus split. The companion test currently locks the incorrect behavior in by asserting that the retarget-enabled branch still returns `0x1f00_ffff`.
**Fix:**
```rust
let interval = difficulty_adjustment_interval(consensus_params);
if i64::from(context.height) % interval == 0 {
    if consensus_params.no_pow_retargeting {
        return context.previous_header.bits;
    }

    let Some(anchor) = context.maybe_retarget_anchor.as_ref() else {
        return consensus_params.pow_limit_bits;
    };
    let actual_timespan = (i64::from(context.previous_header.time) - i64::from(anchor.time))
        .clamp(
            consensus_params.pow_target_timespan_seconds / 4,
            consensus_params.pow_target_timespan_seconds * 4,
        );
    return retarget_compact(
        context.previous_header.bits,
        actual_timespan,
        consensus_params.pow_target_timespan_seconds,
        consensus_params.pow_limit_bits,
    );
}
```
Extend `BlockValidationContext` with the data needed to locate the retarget anchor, then update the test to assert the computed retarget result instead of the previous bits.

## Warnings

### WR-01: Reorg And Disconnect Mutate Live Chainstate Before Success Is Known

**File:** `packages/open-bitcoin-chainstate/src/engine.rs:150-172`, `packages/open-bitcoin-chainstate/src/engine.rs:176-202`, `packages/open-bitcoin-node/src/chainstate.rs:103-113`  
**Issue:** `disconnect_tip()` removes the undo record up front and then mutates `self.utxos` in place while unwinding the block. `reorg()` applies those disconnects directly to the live chainstate before validating the replacement branch. If any later disconnect/connect step fails, the method returns an error with the in-memory chain partially mutated. `ManagedChainstate::reorg()` only persists on success, so the in-memory state and stored snapshot can diverge until restart.
**Fix:**
```rust
pub fn reorg(...) -> Result<ChainTransition, ChainstateError> {
    let mut candidate = self.clone();
    let mut transition = ChainTransition::default();

    for block in disconnect_blocks {
        transition.disconnected.push(candidate.disconnect_tip(block)?);
    }
    for anchored_block in replacement_branch {
        transition.connected.push(candidate.connect_block(
            &anchored_block.block,
            anchored_block.chain_work,
            verify_flags,
            consensus_params,
        )?);
    }

    *self = candidate;
    Ok(transition)
}
```
Also change `disconnect_tip()` to inspect the undo entry with `get()` first and only `remove()` it after the unwind succeeds, so a failed disconnect does not destroy recovery data.

### WR-02: Default Local Connect Paths Bypass Future-Timestamp Validation

**File:** `packages/open-bitcoin-chainstate/src/engine.rs:60-67`, `packages/open-bitcoin-node/src/chainstate.rs:57-70`, `packages/open-bitcoin-node/src/network.rs:159-170`, `packages/open-bitcoin-chainstate/src/engine/tests.rs:380-409`, `packages/open-bitcoin-node/src/network/tests.rs:298-331`  
**Issue:** The convenience `connect_block()` wrappers pass `i64::from(block.header.time)` as `current_time`. That makes the contextual future-time check compare the candidate timestamp against itself plus two hours, so locally connected blocks never trip `time-too-new`. The new tests only prove the explicit `_with_current_time` and inbound-message paths reject future blocks; the public convenience APIs still opt out of that rule.
**Fix:**
```rust
pub fn connect_block(
    &mut self,
    block: &Block,
    chain_work: u128,
    current_time: i64,
    verify_flags: ScriptVerifyFlags,
    consensus_params: ConsensusParams,
) -> Result<ChainPosition, ChainstateError> {
    self.connect_block_with_current_time(
        block,
        chain_work,
        current_time,
        verify_flags,
        consensus_params,
    )
}
```
Require callers to supply a real clock value, or inject a clock into the wrapper layer. If the self-timestamped behavior is intentionally test-only, rename it to make the bypass explicit and keep production call sites on `_with_current_time`.
