# Phase 3.3: Taproot and Tapscript Execution - Context

**Gathered:** 2026-04-12  
**Status:** Ready for planning and execution  
**Mode:** Yolo

<domain>
## Phase Boundary

Phase 3.3 extends the canonical transaction-aware verifier from the completed
legacy and segwit-v0 surface into taproot key-path and tapscript script-path
execution. It owns taproot sighash computation, Schnorr verification, control
block or tapleaf validation, annex handling, tapscript-only opcode behavior,
and taproot discouragement or upgrade flags. It does not own the full imported
parity corpus; that remains Phase 3.4 work.

</domain>

<decisions>
## Implementation Decisions

### Verifier Structure
- **D-01:** Keep `verify_input_script` as the single canonical spending-path
  verifier and extend the existing witness-program branch for witness v1
  taproot instead of introducing a parallel taproot verification API.
- **D-02:** Reuse the current `ScriptExecutionData` surface for tapleaf hash,
  codeseparator position, annex presence or hash, and validation weight, adding
  only the minimum missing fields needed for taproot sighash compatibility.

### Cryptography and Sighash
- **D-03:** Use the existing focused `secp256k1` crate for Schnorr verification
  and x-only tweak checks, while keeping taproot sighash, control-block, and
  tapleaf logic first-party in `open-bitcoin-consensus`.
- **D-04:** Implement BIP341 or BIP342 taproot sighash semantics directly in
  the repo-owned consensus code rather than introducing another dependency.

### Taproot Scope
- **D-05:** Support taproot key-path spends and tapscript script-path spends for
  the current leaf version `0xc0`, including annex handling and control-block
  verification.
- **D-06:** Treat unknown taproot leaf versions and unknown pubkey types through
  the same soft-fork-compatible discouragement points as Knots rather than
  guessing new semantics.

### Tapscript Semantics
- **D-07:** Add only the tapscript-specific opcode behavior needed for this
  phase, especially `CHECKSIGADD` and the tapscript version of `CHECKSIG`
  handling, without expanding the legacy opcode surface further.
- **D-08:** Enforce tapscript validation-weight accounting and tapscript
  `MINIMALIF` in the same execution points as Knots.

### Verification Strategy
- **D-09:** Prefer focused first-party tests for key-path, script-path,
  annex/control-block failures, `CHECKSIGADD`, and taproot discouragement flags
  rather than importing the full upstream taproot corpus in this phase.

</decisions>
