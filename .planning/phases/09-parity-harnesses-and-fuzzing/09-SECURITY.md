# Phase 09 - Security

Per-phase security contract: threat register, accepted risks, and audit trail.

## Scope

Phase 09 delivers verification infrastructure: a first-party test harness crate,
target-neutral RPC functional cases, an Open Bitcoin JSON-RPC black-box parity
suite, optional Knots-compatible RPC execution, integration isolation helpers,
deterministic property-style codec and network tests, parity report emission,
CI report artifact upload, and parity catalog documentation.

No explicit `<threat_model>` block or `## Threat Flags` entries were present in
the Phase 09 plan and summary artifacts. This report therefore records an empty
declared threat register and verifies the security-relevant controls implied by
the phase artifacts and repo guardrails.

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| Test harness crate to verification callers | Reusable test-only helpers drive functional suites, reports, isolation, and HTTP targets without becoming a production RPC dependency. | Case names, method names, params, expected result keys, suite outcomes, report metadata, and HTTP target configuration. |
| Functional cases to Open Bitcoin RPC transport | Black-box parity cases exercise the authenticated JSON-RPC HTTP router instead of bypassing through internal dispatch helpers. | Basic auth credentials from test fixtures, JSON-RPC method names, params, result objects, and error codes. |
| Local test environment to optional Knots endpoint | External Knots-compatible execution is only enabled when all required RPC environment variables are set. | `OPEN_BITCOIN_KNOTS_RPC_ADDR`, `OPEN_BITCOIN_KNOTS_RPC_USER`, `OPEN_BITCOIN_KNOTS_RPC_PASSWORD`, and skipped-suite metadata. |
| Parity report writer to filesystem | Test reports are written only when `OPEN_BITCOIN_PARITY_REPORT_DIR` is configured by tests, `verify.sh`, or CI. | Suite name, target name, pass/fail booleans, case details, skip reason, JSON files, and Markdown files. |
| Integration helpers to host resources | Future daemon-backed tests reserve temporary directories, localhost ports, and child process handles through drop-owned helpers. | Temporary paths, loopback socket addresses, and child process handles. |
| Deterministic generators to parsers and protocol codecs | Generated property-style inputs exercise codec and network boundaries under normal Cargo tests. | Transactions, message headers, wire messages, checksums, generated byte sequences, and malformed/truncated inputs. |
| Repo verification and CI to artifacts | The repo-native verification path creates parity reports and CI uploads them as non-blocking artifacts after the blocking `verify.sh` command. | Report directories, CI artifact paths, Bazel smoke targets, and verification logs. |
| Parity docs to reviewers | Catalog entries describe implemented verification coverage and deferred harness/fuzzing capabilities. | Human-readable coverage claims, Knots source anchors, first-party implementation paths, and known gaps. |

## Controls Verified

| Control | Evidence | Status |
|---------|----------|--------|
| First-party test harness ownership | `packages/Cargo.toml` includes `open-bitcoin-test-harness`; `BUILD.bazel` exposes `//:test_harness`; `scripts/verify.sh` includes `//:test_harness` in the Bazel smoke build. | verified |
| Production dependency boundary | `open-bitcoin-rpc` depends on `open-bitcoin-test-harness` only as a dev-dependency, and Phase 09 did not widen node, wallet, RPC, or CLI runtime behavior. | verified |
| Authenticated black-box RPC boundary | `black_box_parity.rs` starts the real Open Bitcoin RPC HTTP router with explicit test credentials and runs cases through `RpcHttpTarget` instead of internal dispatch helpers. | verified |
| Optional external Knots gating | The Knots-compatible suite requires `OPEN_BITCOIN_KNOTS_RPC_ADDR`, `OPEN_BITCOIN_KNOTS_RPC_USER`, and `OPEN_BITCOIN_KNOTS_RPC_PASSWORD`; missing config records a skipped suite rather than failing local verification or guessing credentials. | verified |
| Report output hygiene | Report writing is opt-in through `OPEN_BITCOIN_PARITY_REPORT_DIR`, writes only suite status/case metadata, sanitizes suite and target names for filenames, and does not serialize RPC passwords or Authorization headers. | verified |
| Integration resource isolation | `Sandbox`, `PortReservation`, and `ProcessGuard` allocate unique temp paths, reserve loopback ports, and clean up child processes on drop for future daemon-backed parity tests. | verified |
| Deterministic codec property coverage | Codec property tests use deterministic generated inputs to prove transaction and message-header round trips and truncated-header rejection without nondeterministic fuzz seeds. | verified |
| Deterministic network property coverage | Network property tests prove wire-message round trips and checksum mutation rejection using deterministic generated messages. | verified |
| Repo-native verification contract | `scripts/verify.sh` defaults `OPEN_BITCOIN_PARITY_REPORT_DIR`, runs the parity report path under normal workspace tests, and remains the CI blocking command. | verified |
| CI artifact handling | `.github/workflows/ci.yml` uploads parity reports with `if: always()` and `if-no-files-found: ignore`, preserving report visibility without weakening the blocking verification step. | verified |
| Honest parity documentation | `docs/parity/catalog/verification-harnesses.md`, `docs/parity/catalog/README.md`, and `docs/parity/index.json` document implemented harness/property coverage and known gaps such as full Knots process lifecycle and dedicated libFuzzer runtime. | verified |
| Unsafe-code and panic-site guardrails | `open-bitcoin-test-harness` forbids unsafe code at the crate root, and `scripts/check-panic-sites.sh` found no unclassified production panic-like sites. | verified |
