<domain>
## Phase Boundary

Phase 07.3 is a narrow maintainability insertion between the completed Phase
07.2 cleanup and the upcoming Phase 8 interface work. It does not add new CLI,
RPC, wallet, or networking features, and it does not widen into a generic
correctness-hardening or style-only rewrite. It only:

1. flattens the highest-value deeply nested control-flow hotspots in
   first-party Rust production code,
2. prefers guard clauses, `let...else`, and small helper extraction where they
   make the happy path easier to audit,
3. preserves the existing external behavior, public crate surfaces, and parity
   claims before Phase 8 planning builds on these crates.

</domain>

## Implementation Decisions

### Scope and targeting
- **D-01:** Limit the phase to the highest-value first-party Rust production
  hotspots surfaced by the scout, starting with
  `packages/open-bitcoin-consensus/src/script/legacy.rs`,
  `packages/open-bitcoin-consensus/src/block.rs`,
  `packages/open-bitcoin-mempool/src/pool.rs`,
  `packages/open-bitcoin-network/src/peer.rs`,
  `packages/open-bitcoin-chainstate/src/engine.rs`, and
  `packages/open-bitcoin-wallet/src/address.rs`. Confirm the final edit list
  during planning instead of turning this into a repo-wide sweep.
- **D-02:** Keep vendored Knots code, first-party test-only files, and low-value
  cosmetic restyling out of scope unless a touched production path needs a
  narrow test update to preserve behavior.

### Flattening strategy
- **D-03:** Prefer guard-clause and early-return refactors only where they make
  the main path clearer. Do not mechanically rewrite protocol or state-machine
  `match` branches when the existing branching shape is already the clearest
  representation.
- **D-04:** When nesting comes from optional or fallible extraction, prefer
  Rust-native guard constructs such as `let...else`. When nesting reflects
  mixed responsibilities, extract small helpers or narrower pure functions
  instead of stacking more local branches.
- **D-05:** Preserve the stable module roots and public surfaces established in
  earlier phases. This is a readability pass, not another module-boundary
  reorganization or an illegal-state redesign campaign.

### Verification and completion
- **D-06:** Use targeted unit tests and crate-level checks as iteration aids
  where reordered guards or helper extraction could mask behavior changes, but
  do not broaden this phase into a large new regression-suite effort.
- **D-07:** The phase is not complete until `bash scripts/verify.sh` passes
  after the narrowed hotspot sweep.

### the agent's Discretion
- Exact hotspot ordering, helper names, and where to stop flattening once a
  function's clarity stops improving remain at the agent's discretion.
- A nested `match` or loop may remain when it expresses the protocol or state
  machine more clearly than a forced sequence of guard clauses.

### Folded Todos
- **Reduce nesting with early returns** — This todo is the direct driver for
  Phase 07.3, but the scope is intentionally narrowed to the highest-value
  production hotspots before Phase 8 rather than a blanket repo-wide rewrite.
