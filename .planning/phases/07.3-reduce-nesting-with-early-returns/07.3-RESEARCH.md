### Flattening strategy
- **D-03:** Prefer guard-clause and early-return refactors only where they make
  the main path clearer. Do not mechanically rewrite protocol or state-machine
  `match` branches when the existing branching shape is already the clearest
  representation.
- **D-04:** When nesting comes from optional or fallible extraction, prefer
  Rust-native guard constructs such as `let...else`. When nesting reflects
  mixed responsibilities, extract small helpers or narrower pure functions
  instead of stacking more local branches.
- **D-05:** Preserve the stable module roots and public surfaces established in
  earlier phases. This is a readability pass, not another module-boundary
  reorganization or an illegal-state redesign campaign.

### Verification and completion
- **D-06:** Use targeted unit tests and crate-level checks as iteration aids
  where reordered guards or helper extraction could mask behavior changes, but
  do not broaden this phase into a large new regression-suite effort.
- **D-07:** The phase is not complete until `bash scripts/verify.sh` passes
  after the narrowed hotspot sweep.

### Pattern 1: Guard Before Main Mutation
**What:** Exit early on absent or unusable state before touching the main branch. [CITED: ../coding-and-architecture-requirements/standards/core/code-shape.md][CITED: ../coding-and-architecture-requirements/standards/languages/rust.md]
**When to use:** A single `Option`, `Result`, or validation rule gates the rest of the function. [CITED: https://doc.rust-lang.org/book/ch06-03-if-let.html]
**Example:**
```rust
// Source: packages/open-bitcoin-wallet/src/wallet/scan.rs
let Some(descriptor) = wallet.descriptor(utxo.descriptor_id) else {
    return false;
};
if !descriptor.descriptor.can_sign() {
    return false;
}
```
[VERIFIED: packages/open-bitcoin-wallet/src/wallet/scan.rs]

### Pattern 2: Use Short Let-Chains for Short-Circuitable Presence Checks
**What:** Combine one or two dependent presence checks when the combined condition is still easier to read than nested `if let` blocks. [CITED: https://doc.rust-lang.org/edition-guide/rust-2024/let-chains.html][CITED: https://doc.rust-lang.org/reference/expressions/if-expr.html?highlight=condition]
**When to use:** The branch is still short enough that the reader can hold the whole condition in working memory. [CITED: https://doc.rust-lang.org/reference/expressions/if-expr.html?highlight=condition]
**Example:**
```rust
// Source: packages/open-bitcoin-consensus/src/block.rs
if context.consensus_params.enforce_segwit
    && let Some(commitment_index) = maybe_commitment_index
{
    // ...
}
```
[VERIFIED: packages/open-bitcoin-consensus/src/block.rs]

### Pattern 3: Extract Small Helpers, Keep Protocol Matches
**What:** Move repeated lookup or cleanup logic into a small private helper while leaving a true protocol or state-machine `match` in place. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][CITED: ../coding-and-architecture-requirements/standards/core/code-shape.md]
**When to use:** The repeated branch work is boilerplate, but the outer `match` still documents wire-protocol or opcode semantics clearly. [VERIFIED: packages/open-bitcoin-network/src/peer.rs][VERIFIED: packages/open-bitcoin-consensus/src/script/legacy.rs]
**Example:** `handle_message` in `peer.rs` is a good fit for extracting repeated peer lookup or request-state cleanup, but not for replacing the top-level `WireNetworkMessage` dispatch with a different control-flow model. [VERIFIED: packages/open-bitcoin-network/src/peer.rs]

### Anti-Patterns to Avoid
- **Mechanical flattening of protocol `match` arms:** `WireNetworkMessage` dispatch in `peer.rs` and opcode dispatch or sig-version branching in `legacy.rs` are often the clearest representation of the underlying protocol state. [VERIFIED: packages/open-bitcoin-network/src/peer.rs][VERIFIED: packages/open-bitcoin-consensus/src/script/legacy.rs][VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md]
- **Smuggling in another refactor wave:** Reorganizing module boundaries, widening visibility, or redesigning types would repeat Phase 07.1 rather than finishing 07.3. [VERIFIED: .planning/phases/07.1-codebase-maintainability-refactor-wave/07.1-CONTEXT.md][VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md]
- **Turning this into a panic sweep:** `address.rs` still has `push_data` panicking on oversize pushes, but the context explicitly defers panic and illegal-state hardening to another phase. [VERIFIED: packages/open-bitcoin-wallet/src/address.rs][VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md]

## Common Pitfalls

### Pitfall 1: Flattening the Wrong Branches
**What goes wrong:** A plan tries to flatten large `match`-driven protocol or state-machine code just because it is nested. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][VERIFIED: packages/open-bitcoin-network/src/peer.rs][VERIFIED: packages/open-bitcoin-consensus/src/script/legacy.rs]
**Why it happens:** The code-shape rule prefers early returns, but it also explicitly allows nesting when it is the clearer representation of the problem. [CITED: ../coding-and-architecture-requirements/standards/core/code-shape.md]
**How to avoid:** Treat opcode dispatch, wire-message dispatch, and sig-version branching as keep-by-default unless a smaller local helper can remove only the repetitive boilerplate around them. [VERIFIED: packages/open-bitcoin-consensus/src/script/legacy.rs][VERIFIED: packages/open-bitcoin-network/src/peer.rs]
**Warning signs:** The diff replaces a single `match` with many tiny helpers, or it becomes harder to see which protocol branch owns which behavior. [VERIFIED: packages/open-bitcoin-consensus/src/script/legacy.rs][VERIFIED: packages/open-bitcoin-network/src/peer.rs]

### Pitfall 2: Widening Into Another Maintainability Phase
**What goes wrong:** A readability pass starts absorbing panic cleanup, interface work, module-boundary changes, or cosmetic restyling. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md]
**Why it happens:** The nominated hotspot files overlap with other deferred cleanup ideas, especially `address.rs` and other large files. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][VERIFIED: packages/open-bitcoin-wallet/src/address.rs]
**How to avoid:** Add explicit stop conditions to the plan: no CLI/RPC work, no panic sweep, no new module split, and no vendored-code edits. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md]
**Warning signs:** The task list starts mentioning `panic!`, `expect!`, CLI parity, or public-surface changes rather than control-flow clarity. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][VERIFIED: packages/open-bitcoin-wallet/src/address.rs]

### Pitfall 3: Over-Extracting Micro-Helpers
**What goes wrong:** A function becomes a sequence of opaque one-line helpers that hide the happy path instead of clarifying it. [CITED: ../coding-and-architecture-requirements/standards/core/code-shape.md]
**Why it happens:** Flattening pressure can tempt the planner to trade indentation for indirection even when the logic is only a few lines long. [CITED: ../coding-and-architecture-requirements/standards/core/code-shape.md]
**How to avoid:** Prefer helper extraction only when the helper has a stable, meaningful responsibility, such as peer lookup or repeated error mapping. [VERIFIED: packages/open-bitcoin-network/src/peer.rs][VERIFIED: packages/open-bitcoin-consensus/src/block.rs]
**Warning signs:** Helper names become generic, the function body reads like a checklist of wrappers, or every helper has only one call site and no standalone meaning. [CITED: ../coding-and-architecture-requirements/standards/core/code-shape.md]

### Pitfall 4: Treating Targeted Checks as the Done Gate
**What goes wrong:** The work looks green at crate scope but fails the repo-native verify flow later. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][VERIFIED: scripts/verify.sh]
**Why it happens:** The phase allows targeted tests as iteration aids, but the completion gate remains `bash scripts/verify.sh`. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][VERIFIED: AGENTS.md]
**How to avoid:** Bake both levels into the plan: quick crate-level feedback while editing, then full repo verification before phase completion. [VERIFIED: .planning/phases/07.3-reduce-nesting-with-early-returns/07.3-CONTEXT.md][VERIFIED: scripts/verify.sh]
**Warning signs:** The plan has test commands for individual crates but no final verify step, or it assumes formatting and Bazel checks are covered elsewhere. [VERIFIED: scripts/verify.sh]
