## Critical Issues

### CR-01: Contextual Header Validation Ignores Difficulty And Future-Time Rules

**File:** `packages/open-bitcoin-consensus/src/block.rs:166-178`
**Issue:** `check_block_header_contextual()` only rejects timestamps at or below median-time-past. It never uses `context.previous_header`, and `BlockValidationContext` does not carry any current-time input, so callers accept headers whose self-declared `bits` merely satisfy their own proof-of-work target and headers that are far in the future. Knots rejects both cases in `ContextualCheckBlockHeader()` by comparing `nBits` against the next required work and by enforcing `time-too-new`. In this repo, `packages/open-bitcoin-chainstate/src/engine.rs:77-84` routes block connection through this function, so the gap is on the active validation path.
**Fix:**
```rust
// Add the missing clock / PoW schedule inputs to BlockValidationContext first.
pub fn check_block_header_contextual(
    header: &BlockHeader,
    context: &BlockValidationContext,
) -> Result<(), BlockValidationError> {
    let expected_bits = next_work_required(
        &context.previous_header,
        header,
        &context.consensus_params,
    )?;
    if header.bits != expected_bits {
        return Err(block_error(
            BlockValidationResult::InvalidHeader,
            "bad-diffbits",
            Some("incorrect proof of work".to_string()),
        ));
    }

    if i64::from(header.time) <= context.previous_median_time_past {
        return Err(block_error(
            BlockValidationResult::InvalidHeader,
            "time-too-old",
            Some("block's timestamp is too early".to_string()),
        ));
    }

    if i64::from(header.time) > context.current_time + MAX_FUTURE_BLOCK_TIME {
        return Err(block_error(
            BlockValidationResult::TimeFuture,
            "time-too-new",
            Some("block timestamp too far in the future".to_string()),
        ));
    }

    Ok(())
}
```

### CR-02: Non-Strict ECDSA Verification Rejects Knots-Compatible Lax DER Signatures

**File:** `packages/open-bitcoin-consensus/src/signature.rs:238-248`
**Issue:** `parse_ecdsa_signature_for_verification()` always parses with `ecdsa::Signature::from_der()`. When DER parsing fails and `DERSIG`/`STRICTENC`/`LOW_S` are not enabled, the function returns `Ok(None)`, which makes script evaluation treat the signature as simply false. That does not match Knots: `CPubKey::Verify()` still uses `ecdsa_signature_parse_der_lax()` on non-strict paths and verifies historically accepted pre-BIP66 encodings. This implementation can therefore reject blocks or transactions that Knots accepts.
**Fix:**
```rust
fn parse_ecdsa_signature_for_verification(
    bytes: &[u8],
    flags: ScriptVerifyFlags,
) -> Result<Option<ParsedEcdsaSignature>, SignatureError> {
    let Some((&hash_type, der)) = bytes.split_last() else {
        return Ok(None);
    };
    let sighash_type = SigHashType::from_u32(u32::from(hash_type));

    if flags.contains(ScriptVerifyFlags::STRICTENC) {
        validate_legacy_sighash_type(sighash_type)?;
    }

    let require_strict_der = flags.contains(ScriptVerifyFlags::DERSIG)
        || flags.contains(ScriptVerifyFlags::LOW_S)
        || flags.contains(ScriptVerifyFlags::STRICTENC);

    let signature = if require_strict_der {
        ecdsa::Signature::from_der(der).map_err(|_| SignatureError::InvalidDer)?
    } else {
        parse_ecdsa_signature_lax(der).ok_or(SignatureError::IncorrectSignature)?
    };

    // Keep the LOW_S gate after parsing.
    Ok(Some(ParsedEcdsaSignature {
        signature,
        sighash_type,
    }))
}
```
Add a regression test with a Knots-accepted lax-DER signature vector so the non-strict path proves parity instead of codifying the stricter behavior.
