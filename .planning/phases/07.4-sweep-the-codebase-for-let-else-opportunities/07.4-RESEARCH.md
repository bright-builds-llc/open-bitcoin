### Let-else heuristics
- **D-03:** Prefer `let...else` only when a function has a focused guard-style
  extraction from `Option`, `Result`, or pattern-matched input and the `else`
  branch is an immediate `return`, `continue`, `break`, or typed error path.
  The goal is to move the happy path to the left margin, not to maximize the
  number of syntax conversions.
- **D-04:** Keep `if let`, `match`, or plain conditionals when they express
  optional side effects, ordered recognizer ladders, or genuine multi-branch
  protocol logic more clearly than `let...else`. In particular,
  `packages/open-bitcoin-wallet/src/descriptor.rs` and
  `packages/open-bitcoin-consensus/src/classify.rs` should only change if the
  rewrite is plainly easier to audit than the current ordered flow.

## Summary

This phase should be planned as an audit-first readability pass, not as a quota-driven rewrite. The repo rules, the 07.4 context, and the official Rust guidance all align on the same heuristic: use `let...else` when it cleanly expresses guard-style extraction with a diverging `else`, and keep `if let` or `match` when the branch is a normal success path, an ordered recognizer ladder, or an optional side effect. ### Anti-Patterns to Avoid

- **Mechanical rewrite of every `if let` hit:** The 07.4 context explicitly rejects a repo-wide mechanical sweep, and the code audit shows many remaining hits are better left as-is. [VERIFIED: .planning/phases/07.4-sweep-the-codebase-for-let-else-opportunities/07.4-CONTEXT.md][VERIFIED: codebase rg 2026-04-19]
- **Using `let...else` when the `else` branch is a normal success path:** `let...else` is for divergent guards, not for “either genesis or parent” or “either change output or none” success branches. [CITED: https://doc.rust-lang.org/beta/reference/statements.html][VERIFIED: packages/open-bitcoin-network/src/header_store.rs][VERIFIED: packages/open-bitcoin-wallet/src/wallet/build.rs]
- **Replacing ordered recognizer ladders with nested guard exits:** That makes classifiers and descriptor parsers harder to audit, not easier. [VERIFIED: packages/open-bitcoin-consensus/src/classify.rs][VERIFIED: packages/open-bitcoin-wallet/src/descriptor.rs]
- **Introducing helper functions purely to force `let...else`:** The context allows narrow helpers when responsibilities are mixed, but not helper sprawl just to satisfy a syntax preference. [VERIFIED: .planning/phases/07.4-sweep-the-codebase-for-let-else-opportunities/07.4-CONTEXT.md]

## Common Pitfalls

### Pitfall 1: Mechanical Conversion of Ordered Ladders

**What goes wrong:** Parser and classifier ladders become harder to follow because a straightforward ordered search is turned into a series of guard exits. [VERIFIED: packages/open-bitcoin-wallet/src/descriptor.rs][VERIFIED: packages/open-bitcoin-consensus/src/classify.rs]  
**Why it happens:** `if let` hits look superficially convertible even when each rung is part of a recognizer sequence instead of a guard for the rest of the function. [VERIFIED: code inspection 2026-04-19]  
**How to avoid:** Keep the current ladder unless the rewrite is plainly easier to audit than the existing order. [VERIFIED: .planning/phases/07.4-sweep-the-codebase-for-let-else-opportunities/07.4-CONTEXT.md]  
**Warning signs:** Multiple `return` statements in a parser ladder, or a rewrite that makes the valid descriptor/script forms less obvious. [VERIFIED: packages/open-bitcoin-wallet/src/descriptor.rs][VERIFIED: packages/open-bitcoin-consensus/src/classify.rs]

### Pitfall 2: Forcing `let...else` onto Optional Side Effects

**What goes wrong:** A small optional write turns into a longer branch or a duplicated code path. [VERIFIED: packages/open-bitcoin-wallet/src/address.rs][VERIFIED: packages/open-bitcoin-wallet/src/wallet/sign.rs][VERIFIED: packages/open-bitcoin-mempool/src/error.rs]  
**Why it happens:** The code uses `if let` to say “do this extra thing when present,” not “I need this value for the rest of the function.” [VERIFIED: code inspection 2026-04-19]  
**How to avoid:** Keep `if let` for optional payload appends, redeem-script writes, and display-format branches. [VERIFIED: packages/open-bitcoin-wallet/src/address.rs][VERIFIED: packages/open-bitcoin-wallet/src/wallet/sign.rs][VERIFIED: packages/open-bitcoin-mempool/src/error.rs]  
**Warning signs:** The converted `else` would return `Ok(())`, `None`, or some other normal success case instead of an actual guard exit. [CITED: https://doc.rust-lang.org/beta/reference/statements.html]

### Pitfall 3: Using `let...else` When the Non-Match Path Is Still Success

**What goes wrong:** The rewrite inverts the mental model and often duplicates work because the non-match branch is not an error at all. [VERIFIED: packages/open-bitcoin-network/src/header_store.rs][VERIFIED: packages/open-bitcoin-wallet/src/wallet/build.rs]  
**Why it happens:** `let...else` requires a diverging `else`, but some scout hits represent alternate successful flows like genesis handling or “no change output.” [CITED: https://doc.rust-lang.org/beta/reference/statements.html][VERIFIED: packages/open-bitcoin-network/src/header_store.rs][VERIFIED: packages/open-bitcoin-wallet/src/wallet/build.rs]  
**How to avoid:** Leave those branches as `if let` or `match`, or restructure only if the whole function becomes simpler without duplicate insertion or output-building logic. [VERIFIED: code inspection 2026-04-19]  
**Warning signs:** You are tempted to use `let None = ... else { ... }`, add `unreachable!()`, or duplicate the rest of the function into an `else`. [VERIFIED: code inspection 2026-04-19]

### Pitfall 4: Extending Borrow and Binding Scope Accidentally

**What goes wrong:** A converted binding lives for the rest of the surrounding block, which can widen borrow scope or change when a temporary is dropped. [CITED: https://doc.rust-lang.org/beta/reference/statements.html][CITED: https://doc.rust-lang.org/nightly/edition-guide/rust-2024/temporary-if-let-scope.html]  
**Why it happens:** `let...else` binds in outer scope, while Rust 2024 also changed `if let` scrutinee temporary scope relative to older editions. [VERIFIED: packages/Cargo.toml][CITED: https://doc.rust-lang.org/beta/reference/statements.html][CITED: https://doc.rust-lang.org/nightly/edition-guide/rust-2024/temporary-if-let-scope.html]  
**How to avoid:** Review converted sites that borrow from maps, slices, or temporary expressions, and keep `if let` or `match` if the wider scope hurts clarity or ownership. [VERIFIED: code inspection 2026-04-19]  
**Warning signs:** A rewrite introduces extra clones, borrow-checker friction, or a need to shorten scope manually with nested blocks. [VERIFIED: code inspection 2026-04-19]
