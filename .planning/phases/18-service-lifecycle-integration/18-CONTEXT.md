# Phase 18: Service Lifecycle Integration - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning
**Mode:** Yolo

<domain>
## Phase Boundary

Implement macOS launchd and Linux systemd service lifecycle support — install, uninstall, enable, disable, and status — wired into the existing `ServiceCommand` clap surface from Phase 13. Phase 18 makes those commands functional with dry-run-first safety, privileged-action surfacing, and isolated test coverage. This phase does not build the Ratatui dashboard (Phase 19), expand wallet runtime behavior (Phase 20), implement migration mutation (Phase 21), or add real-sync benchmarks (Phase 22).

</domain>

<decisions>
## Implementation Decisions

### Service Manager Architecture
- **D-01:** Implement a `ServiceManager` trait in `open-bitcoin-node/src/service.rs` with `install`, `uninstall`, `enable`, `disable`, and `status` methods returning typed results. No third-party service manager crates — implement first-party `LaunchdManager` and `SystemdManager` structs that invoke `launchctl` and `systemctl` as subprocesses through the `std::process::Command` adapter shell.
- **D-02:** Platform detection uses compile-time `#[cfg(target_os = "macos")]` / `#[cfg(target_os = "linux")]` guards in the CLI adapter to select the concrete manager. The trait itself is platform-agnostic so pure tests can use a `FakeServiceManager` without any OS guard.
- **D-03:** macOS is the higher-priority platform. Linux systemd support must fit the same `ServiceManager` trait surface without changing the public command interface.

### Plist and Unit File Generation
- **D-04:** macOS plist XML and Linux systemd unit file text are generated by pure helper functions in `open-bitcoin-node/src/service.rs` that take typed `ServiceConfig` inputs and return `String`. These helpers are exercised in unit tests without any filesystem or subprocess access.
- **D-05:** `ServiceConfig` captures: daemon binary path, config file path, log output path, error output path, keep-alive policy, and run-at-load flag. These become the authoritative inputs to both plist and unit file renderers.
- **D-06:** The generated plist label uses a deterministic reverse-domain label (`org.open-bitcoin.node`). The systemd unit name uses `open-bitcoin.service`. Both are defined as constants.

### Dry-Run and Privilege Safety
- **D-07:** All mutating service commands (`install`, `uninstall`, `enable`, `disable`) default to a dry-run preview mode. The preview shows generated file contents, proposed filesystem paths, required privilege level, daemon command, config path, log path, and recovery behavior before any write occurs.
- **D-08:** Require an explicit `--apply` flag on the `ServiceArgs` struct to execute real changes. Without `--apply`, the command renders the plan and exits with success. This matches the operator-trust philosophy from Phase 17.
- **D-09:** Commands must state privilege requirements before acting: macOS `install`/`uninstall` typically requires `sudo` for system-level launchd (`/Library/LaunchDaemons`); user-level install (`~/Library/LaunchAgents`) does not. Linux systemd requires `sudo` for system unit installation.

### Service Status
- **D-10:** `service status` queries the real service manager subprocess (`launchctl list org.open-bitcoin.node` / `systemctl is-active open-bitcoin.service`) and maps the output to `ServiceStatus` states: `installed`, `enabled`, `running`, `failed`, `stopped`, `unmanaged`.
- **D-11:** The `ServiceStatus` model from Phase 13 (`manager`, `installed`, `enabled`, `running`) is extended to explicitly represent `failed` and `unmanaged` states so status and dashboard consumers have the full picture.
- **D-12:** When the service manager subprocess is unavailable (e.g., running on an unsupported OS or without the manager binary), the status should return `unmanaged` with a reason rather than returning an error.

### Testing Approach
- **D-13:** All tests run against isolated temp paths or a `FakeServiceManager` trait implementation. No test may call real `launchctl` or `systemctl` or modify the developer machine's actual service registrations.
- **D-14:** Pure plist and unit file generation helpers are unit-tested without process isolation. The `FakeServiceManager` is tested for correct state tracking. CLI adapter integration tests use temp directories and the fake manager.
- **D-15:** Binary integration tests in `tests/operator_binary.rs` may test `service status` and dry-run `service install` using the `FakeServiceManager` injection path or by asserting on rendered output that includes "dry-run" or "apply" markers.

### Module and File Location
- **D-16:** Service trait, pure generators, `ServiceConfig`, and `ServiceStatus` extensions live in `packages/open-bitcoin-node/src/service.rs` (and `service/` subdirectory if it grows beyond ~300 lines).
- **D-17:** CLI adapter with subprocess execution lives in `packages/open-bitcoin-cli/src/operator/service.rs`. The `execute_operator_cli_inner` match arm for `OperatorCommand::Service` delegates to this adapter instead of returning the "deferred to Phase 18" error.
- **D-18:** `ServiceArgs` gains an `--apply` flag and optionally a `--user` flag to select user-level vs system-level service scope on macOS. No new `ServiceArgs` fields that change the existing command shape.

### Claude's Discretion
- Exact field names, helper method signatures, and Rust module structure within the described boundaries are discretionary.
- The first macOS implementation may use a user-level (`~/Library/LaunchAgents`) install path as the default before adding system-level support, provided the CLI clearly communicates which scope is active.
- Linux systemd support may be implemented as a stub returning `unmanaged` on macOS builds and fully functional on Linux builds, provided the trait, tests, and dry-run output work the same way on both.

</decisions>

<specifics>
## Specific Ideas

- Favor explicit over implicit: every mutating command must say what it will do and where before doing it. The `--apply` flag is the operator's explicit consent.
- Quiet and information-dense output style from Phase 17 applies: show paths, scopes, and states without marketing prose.
- macOS user-level install (`~/Library/LaunchAgents`) is the safer default for initial implementation because it does not require `sudo`. System-level install (`/Library/LaunchDaemons`) can follow with explicit privilege surfacing.
- `launchctl` label `org.open-bitcoin.node` and systemd unit `open-bitcoin.service` should be derivable from a single constant, not hardcoded in multiple places.

</specifics>
