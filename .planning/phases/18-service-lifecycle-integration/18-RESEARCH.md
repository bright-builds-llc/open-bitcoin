- **D-12:** When service manager subprocess is unavailable, return `unmanaged` with reason (not an error).
- **D-13:** All tests use isolated temp paths or `FakeServiceManager`. No real `launchctl`/`systemctl` calls in tests.
- **D-14:** Pure generators are unit-tested without process isolation. `FakeServiceManager` tested for correct state tracking. CLI adapter integration tests use temp dirs and fake manager.

## Summary

Phase 18 wires the already-scaffolded `ServiceCommand` variants (`Install`, `Uninstall`, `Enable`, `Disable`, `Status`) into real implementations for macOS launchd and Linux systemd. The architecture follows three clean layers: (1) a platform-agnostic `ServiceManager` trait with a `FakeServiceManager` for tests, (2) pure text generator functions for plist and systemd unit content (testable without I/O), and (3) concrete `LaunchdManager` and `SystemdManager` subprocess adapters injected at CLI call sites.

The key safety mechanism is the `--apply` flag: without it, all mutating commands render a dry-run plan showing exactly what files would be written, what privilege is needed, and what the recovery behavior is. The operator confirms by passing `--apply`. This pattern mirrors the onboarding `--approve-write` flag already established in Phase 17.

macOS `launchctl` is confirmed available on the developer machine (Darwin 25.3.0, launchctl bootstrapper 7.0.0). The user-level LaunchAgents path (`~/Library/LaunchAgents/`) is the safer default because it requires no `sudo`. A reference plist file is available in the Knots submodule (`contrib/init/org.bitcoin.bitcoind.plist`) as structural reference. The systemd unit file format is also available from `contrib/init/bitcoind.service`.

**Primary recommendation:** Implement `ServiceConfig` and pure generators first (no I/O, easy to test), then `FakeServiceManager` (enables all CLI adapter tests), then `LaunchdManager` subprocess adapter (macOS target), then wire the CLI arm, then add binary integration tests. Add `SystemdManager` as a stub on macOS and full implementation in the same wave.

## Architecture Patterns

### Recommended Module Structure

```
packages/open-bitcoin-node/src/
└── service.rs              # ServiceManager trait, ServiceConfig, ServiceStatus extensions,
                            # pure plist/unit generators
    service/                # (if service.rs grows beyond ~300 lines)
    └── tests.rs            # #[cfg(test)] pure generator tests, FakeServiceManager tests

packages/open-bitcoin-cli/src/operator/
└── service.rs              # execute_service_command(), LaunchdManager, SystemdManager,
                            # ServiceArgs extension, dry-run renderer
```

### Pattern 1: Pure Generator Functions

**What:** Helper functions in `service.rs` that take `ServiceConfig` and return `String` (plist XML or systemd unit text). No filesystem or subprocess access.

**When to use:** All plist and unit file content generation. Also drives the dry-run preview.

**Example plist output format** [VERIFIED: inspected `~/Library/LaunchAgents/homebrew.mxcl.postgresql@17.plist` and `contrib/init/org.bitcoin.bitcoind.plist`]:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>org.open-bitcoin.node</string>
    <key>ProgramArguments</key>
    <array>
        <string>/path/to/open-bitcoin-node</string>
        <string>--config</string>
        <string>/path/to/open-bitcoin.jsonc</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
    <key>StandardOutPath</key>
    <string>/path/to/open-bitcoin.log</string>
    <key>StandardErrorPath</key>
    <string>/path/to/open-bitcoin-error.log</string>
</dict>
</plist>
```

**Example systemd unit format** [VERIFIED: `contrib/init/bitcoind.service`]:

### Pattern 2: ServiceManager Trait and FakeServiceManager

**What:** Trait captures the five lifecycle operations and `status`. `FakeServiceManager` tracks in-memory state for tests. Concrete managers wrap subprocess invocations.

**When to use:** Every CLI adapter call goes through the trait. Tests inject `FakeServiceManager`. Production CLI injects the real manager.

```rust
// Source: pattern derived from StatusRpcClient in operator/status.rs [VERIFIED: codebase]
pub trait ServiceManager {
    fn install(&self, config: &ServiceConfig, scope: ServiceScope) -> Result<(), ServiceError>;
    fn uninstall(&self, scope: ServiceScope) -> Result<(), ServiceError>;
    fn enable(&self, scope: ServiceScope) -> Result<(), ServiceError>;
    fn disable(&self, scope: ServiceScope) -> Result<(), ServiceError>;
    fn status(&self) -> Result<ServiceManagerStatus, ServiceError>;
}
```

### Pattern 3: Dry-Run Preview Before --apply

**What:** All mutating commands (`install`, `uninstall`, `enable`, `disable`) render a plan showing file path, generated content, privilege requirement, daemon command, config path, log path, and recovery behavior. The plan renders the same way whether or not `--apply` is passed. Without `--apply`, the function returns after printing the plan.

**When to use:** Every mutating `ServiceCommand` variant handler.

**Privilege surfacing rules** [VERIFIED: launchctl behavior on developer machine]:
- User-level install: `~/Library/LaunchAgents/org.open-bitcoin.node.plist` — no `sudo` required
- System-level install: `/Library/LaunchDaemons/org.open-bitcoin.node.plist` — requires `sudo`
- Linux user unit: `~/.config/systemd/user/open-bitcoin.service` — no `sudo` required
- Linux system unit: `/etc/systemd/system/open-bitcoin.service` — requires `sudo`

### Pattern 4: launchctl Subprocess Adapter

**Key commands** [VERIFIED: live `launchctl` on macOS Darwin 25.3.0]:

```
Install (user-level):
  1. Write plist to ~/Library/LaunchAgents/org.open-bitcoin.node.plist
  2. launchctl bootstrap gui/<uid> ~/Library/LaunchAgents/org.open-bitcoin.node.plist

Uninstall:
  1. launchctl bootout gui/<uid>/org.open-bitcoin.node
  2. Remove plist file

Enable (persist across reboots):
  launchctl enable gui/<uid>/org.open-bitcoin.node

Disable:
  launchctl disable gui/<uid>/org.open-bitcoin.node

Status query:
  launchctl list org.open-bitcoin.node
  Exit code 0 = service known to launchd
  Exit code 113 = service not found [VERIFIED: live test]
```

**State mapping from `launchctl list` output** [VERIFIED: inspected running service output on developer machine]:
- Exit code 113 -> `Unmanaged` (not installed)
- Exit code 0, `"PID" = ` present in stdout -> `Running`
- Exit code 0, `"PID"` absent, `"LastExitStatus" = 0;` in stdout -> `Stopped`
- Exit code 0, `"PID"` absent, `"LastExitStatus"` non-zero in stdout -> `Failed`
- `launchctl` binary not found -> `Unmanaged` with reason "launchctl not found"

**Critical note on output format:** `launchctl list <label>` produces CF plist text (e.g., `"PID" = 913;`) not JSON. Use string pattern matching, not `serde_json`.

### Pattern 5: ServiceStatus Extended States

The existing `ServiceStatus` struct uses `FieldAvailability<bool>` for `installed`, `enabled`, `running`. Adding a `lifecycle_state` field preserves backward compat while providing the richer signal required by D-11.

**Recommended approach:** Add `lifecycle_state: FieldAvailability<ServiceLifecycleState>` alongside existing fields. Annotate with `#[serde(default)]` so existing consumers do not break.

```rust
// Proposed enum (discretionary per CONTEXT.md)
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
#[serde(rename_all = "snake_case")]
pub enum ServiceLifecycleState {
    #[default]
    Unmanaged,
    Installed,
    Enabled,
    Running,
    Stopped,
    Failed,
}
```

### Anti-Patterns to Avoid

- **Calling real `launchctl`/`systemctl` in tests:** Mutates developer machine and fails on non-macOS CI. Use `FakeServiceManager` for all tests.
- **Hardcoding the label string in multiple places:** Define `OPEN_BITCOIN_LAUNCHD_LABEL: &str = "org.open-bitcoin.node"` and `OPEN_BITCOIN_SYSTEMD_UNIT: &str = "open-bitcoin.service"` as module constants.
- **Parsing `launchctl list` output with serde:** The output format is CF plist text with `=` not JSON `:`. Use exit code plus string matching.
- **Skipping parity breadcrumb comment:** Every new first-party `*.rs` source file needs the top-of-file breadcrumb block and an entry in `docs/parity/source-breadcrumbs.json`. [VERIFIED: AGENTS.md]
- **Writing to real service directories without `--apply`:** The dry-run path must never call `std::fs::write` or subprocess commands for `launchctl`/`systemctl`.
- **`launchctl load`/`unload`:** Deprecated since macOS 10.10. Use `bootstrap`/`bootout` with domain target `gui/<uid>`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| XML escaping in plist | Custom XML escaper | Service label and paths never contain `<>&"'`; use direct string template | Safe for all realistic path inputs |
| launchctl output parsing | serde deserializer | Exit code + string search for `"PID" = ` and `"LastExitStatus" = ` | Output is CF plist text, not JSON |
| Cross-platform service discovery | Third-party crate | `#[cfg(target_os)]` compile-time guards + `FakeServiceManager` for tests | No crate that fits project dependency policy |
| Process privilege escalation | sudo wrapper in Rust | State the privilege requirement in dry-run output; operator runs with `sudo` if needed | Privilege escalation belongs to the operator, not the tool |

**Key insight:** The complexity in service management is not the subprocess calls — it is the state mapping from subprocess output and the safety guarantees around dry-run. Those are solved by the trait + pure renderer pattern, not by libraries.

## Common Pitfalls

### Pitfall 1: launchctl list output is CF plist text, not JSON

**What goes wrong:** Developer tries `serde_json::from_str` on `launchctl list org.open-bitcoin.node` output and gets a parse error because the output uses `"Key" = Value;` syntax.

**Why it happens:** `launchctl list <label>` produces Core Foundation plist text format, not JSON.

**How to avoid:** Use exit code (0 = found, 113 = not found) and string pattern matching. Look for `"PID" = ` and `"LastExitStatus" = ` in stdout.

**Warning signs:** `serde_json::from_str` returning `Err` on launchctl stdout.

[VERIFIED: observed `launchctl list homebrew.mxcl.postgresql@17` output on developer machine]

### Pitfall 2: launchctl load/unload is deprecated; use bootstrap/bootout

**What goes wrong:** Using `launchctl load ~/Library/LaunchAgents/foo.plist` works but is deprecated. Modern macOS (10.10+) uses `launchctl bootstrap gui/$UID <plist-path>`. [ASSUMED: based on Apple launchd documentation trajectory; verified that `bootstrap` works on current Darwin 25.3.0]

**How to avoid:** Use `launchctl bootstrap gui/<uid> <plist-path>` for load and `launchctl bootout gui/<uid>/<label>` for unload.

**Warning signs:** `launchctl: Deprecated. Please use launchctl bootstrap ...` in stderr.

### Pitfall 3: New ServiceStatus fields break existing JSON consumers

**What goes wrong:** Adding required fields to `ServiceStatus` without `#[serde(default)]` breaks deserialization of older status JSON in tests that compare against fixtures.

**How to avoid:** Annotate new `lifecycle_state` field with `#[serde(default)]`. Keep existing `FieldAvailability<bool>` fields for `installed`, `enabled`, `running` unchanged.

**Warning signs:** Test failures in `operator_binary.rs` or `status/tests.rs` after adding `ServiceStatus` fields.

### Pitfall 4: Missing parity breadcrumb registration

**What goes wrong:** `bash scripts/verify.sh` (or pre-commit hook) fails with a breadcrumb check error for the new `service.rs` files.

**Why it happens:** The repo enforces parity breadcrumbs for all `packages/open-bitcoin-*/src/**/*.rs` files via `scripts/check-parity-breadcrumbs.ts`. [VERIFIED: AGENTS.md repo-local guidance, quick task 260425-kzh]

**How to avoid:** Add entries for `packages/open-bitcoin-node/src/service.rs` and `packages/open-bitcoin-cli/src/operator/service.rs` (with `none` breadcrumb reason) to `docs/parity/source-breadcrumbs.json` in the same wave that creates the files.

**Warning signs:** `check-parity-breadcrumbs` script failure in verify or pre-commit output.

### Pitfall 5: Scope and privilege surfacing computed independently

**What goes wrong:** Dry-run output says "no sudo required" but the install actually writes to `/Library/LaunchDaemons` which requires `sudo`.

**How to avoid:** Derive both the target install path AND the privilege requirement from a single `ServiceScope` enum value. Never compute them in separate code paths.

### Pitfall 6: Tests running on non-macOS CI invoke launchctl

**What goes wrong:** A test that directly constructs `LaunchdManager` fails on Linux CI because `launchctl` is not present.

**How to avoid:** All tests must use `FakeServiceManager`. `LaunchdManager` is only constructed in the production CLI path, gated by `#[cfg(target_os = "macos")]`.

### launchctl state detection via exit code and string matching

```rust
// Source: std::process::Command docs [ASSUMED for exact Rust API;
// exit code 113 for missing service [VERIFIED: live test on Darwin 25.3.0]
fn query_launchd_state(label: &str) -> ServiceManagerStatus {
    let Ok(output) = std::process::Command::new("launchctl")
        .arg("list")
        .arg(label)
        .output()
    else {
        return ServiceManagerStatus::Unmanaged {
            reason: "launchctl not available".to_string(),
        };
    };

    // Exit code 113 = label not registered in domain
    if output.status.code() == Some(113) {
        return ServiceManagerStatus::Unmanaged {
            reason: "service not registered with launchd".to_string(),
        };
    }

    let stdout = String::from_utf8_lossy(&output.stdout);
    // CF plist text: "PID" = 1234; (space around =, semicolon at end)
    if stdout.contains("\"PID\" = ") {
        return ServiceManagerStatus::Running;
    }
    if stdout.contains("\"LastExitStatus\" = 0;") {
        return ServiceManagerStatus::Stopped;
    }
    ServiceManagerStatus::Failed
}
```

### FakeServiceManager state tracking pattern

```rust
// Source: pattern from StatusRpcClient trait in operator/status.rs [VERIFIED: codebase]
use std::sync::{Arc, Mutex};

#[derive(Debug, Default)]
struct FakeServiceState {
    installed: bool,
    enabled: bool,
    maybe_last_lifecycle: Option<ServiceLifecycleState>,
}

pub struct FakeServiceManager {
    state: Arc<Mutex<FakeServiceState>>,
}

impl ServiceManager for FakeServiceManager {
    fn install(&self, _config: &ServiceConfig, _scope: ServiceScope) -> Result<(), ServiceError> {
        let mut s = self.state.lock().expect("state lock");
        s.installed = true;
        s.maybe_last_lifecycle = Some(ServiceLifecycleState::Installed);
        Ok(())
    }
    fn status(&self) -> Result<ServiceManagerStatus, ServiceError> {
        let s = self.state.lock().expect("state lock");
        if !s.installed {
            return Ok(ServiceManagerStatus::Unmanaged {
                reason: "not installed".to_string(),
            });
        }
        Ok(ServiceManagerStatus::Stopped)
    }
    // ... other methods follow same pattern
}
```

```rust
// Source: onboarding plan renderer pattern [VERIFIED: onboarding.rs in codebase]
fn render_service_dry_run(
    command: ServiceCommandName,
    config: &ServiceConfig,
    scope: ServiceScope,
    generated_content: &str,

        daemon = config.daemon_binary_path.display(),
        conf = config.config_file_path.display(),
        log = config.log_output_path.display(),
        recovery = if config.keep_alive { "restart on failure" } else { "no restart" },
    )
}
```

## Open Questions (RESOLVED)

1. **`ServiceStatus` extension strategy**
   - What we know: D-11 requires `failed` and `unmanaged` states. Existing struct uses `FieldAvailability<bool>` for `installed`/`enabled`/`running`.
   - What is unclear: Whether to add a new `lifecycle_state: FieldAvailability<ServiceLifecycleState>` field alongside existing fields (additive, no breakage) or to change the existing fields to a richer type (potentially breaking JSON schema for status consumers from Phase 17).
   - Recommendation: Add `lifecycle_state` as a new optional field with `#[serde(default)]` so existing consumers are unaffected.
   - RESOLVED: Plan 18-03 maps `ServiceLifecycleState` variants to the existing `installed`/`enabled`/`running` `FieldAvailability<bool>` fields — additive approach preserves JSON schema compatibility with Phase 17 status consumers.

2. **`--user` flag naming convention**
   - What we know: Default is user-level (no `sudo`). D-18 says `--user` flag is optional.
   - What is unclear: Whether `--user` selects user-level (redundant for default) or whether the flag should instead be `--system` to select elevated scope.
   - Recommendation: Name the flag `--system` to select system-level install. Absence = user-level (the safe default). This makes the elevated-scope selection explicit and matches the "require explicit consent for elevated operations" philosophy.
   - RESOLVED: Plan 18-02 uses `--apply` (not `--user`/`--system`) as the only new flag. System-level scope is deferred per D-10; user-level is unconditionally the default in this phase.

3. **`launchctl enable` behavior when service is not bootstrapped**
   - What we know: `launchctl enable` sets the disabled override independently of whether the service is currently bootstrapped.
   - What is unclear: Whether `enable` succeeds when the plist file exists but the service is not bootstrapped yet.
   - Recommendation: Implement `enable` as a distinct step from `install`; document in dry-run output that `enable` requires the service to be installed first. The `FakeServiceManager` can enforce this ordering in tests.

## Security Domain

Service lifecycle management is local system configuration, not a network security surface. Applicable ASVS categories are narrow.

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No | Service manager runs as current user or root |
| V3 Session Management | No | No sessions |
| V4 Access Control | Yes | Privilege requirement surfaced in dry-run before `--apply`; user-level default avoids root |
| V5 Input Validation | Yes | `ServiceConfig` paths used as separate `Command::arg()` items, not shell-interpolated strings |
| V6 Cryptography | No | No secrets involved |

**Key security note:** Using `std::process::Command::arg()` for each argument separately (not `Command::new("sh").arg("-c").arg(combined_string)`) prevents shell injection. This pattern is required for all `launchctl`/`systemctl` invocations. [VERIFIED: Rust stdlib Command design]
