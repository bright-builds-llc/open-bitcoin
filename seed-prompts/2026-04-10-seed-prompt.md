We will engineer a fully working Bitcoin Node in Rust. These are the general requirements for making it:
We must have as close to full 100% feature parity with the existing bitcoin node reference implementation, which will live inside of a sub directory within the packages directory, perhaps as a git submodule. We will use Bitcoin Knots pinned to version 29.3.knots20260210 as our reference implementation. https://github.com/bitcoinknots/bitcoin/releases/tag/v29.3.knots20260210
We must have 100% unit test coverage of all pure core code.
We must have as close to one-to-one parity in the spirit of all of the tests that currently exist in the bitcoin node reference implementation.
We must follow the philosophy of strong type safety and here is link to an article that explains this more in depth: Parse-don't-validate article: https://www.harudagondi.space/blog/parse-dont-validate-and-type-driven-design-in-rust
We are ok writing all of the Rust implementation in a more type-safe and idiomatic manner than the C++ code, as long as the external api contracts are withheld and aligned with the reference implementation.
We must use the Scott Wlaschin’s architectural pattern of functional pure core and imperative (I/O) shell: https://www.youtube.com/watch?v=P1vES9AgfC4
We very likely want to use bazel/bazelisk because we will have a multi-language repo. We don’t have to convert the Bitcoin reference implementation to bazel, since it is external code, but newer things will and should use bazel and be invokable at the top level of the repo. We may have other packages, for example, a marketing website and a GUI front end. This will be deferred to a later milestone.
We must have a modular code base. For example, we will have a packages directory that will essentially contain all of the source code in a modular fashion.
We must have sensible, reasonable, and strict requirements about correctness within our AGENTS.md file.
We must implement hard checks that enforce not having any I/O in the in the code that is inside of the pure core. We must create some sort of check that enforces this, so that if there are contributors in the future, they will get early signal of this requirement.
We must implement very strict tests, not only unit test, but also integration tests, and perhaps other kinds of tests, that test certain end to end flows and we might need to create a new framework (or research existing popular frameworks) that allows these tests to be both run in parallel and hermetically so that running one test does not interfere with any other test, and/or utilize Docker to isolate different instances of integration test runs.
It would be very nice to have platform independent integration tests such that they are initially run against the Bitcoin reference implementation to verify and lock down behavior, and then, later, they will be run against our Rust port to prove parity.
We must have very strong and very strict pre-commit checks and CI checks with GitHub workflows and GitHub actions.
We should have some set of benchmarks. There should be some basic benchmarks that test our code base and determine performance in various ways, and it would be nice to be able to run these against the Bitcoin reference implementation so that we have a comparative analysis.
We likely want to create a meta-package that keeps track of the progress of our implementation; perhaps a bare bones, static, SolidJS website which outlines and stays in sync with the progress and milestones and features of the project.
We must strongly adhere to the concept of making illegal states unrepresentable.
We must not add dependencies haphazardly; we must be very wary about adding dependencies. We will allow a few, trusted, basic dependencies, for example, tokio for asynchronous programming, rustls for basic TLS functionality and encryption, and other very well maintained, and very popular, trusted crypto libraries. We will not utilize existing Bitcoin rust libraries; in fact, we will architect, engineer, and export our own Bitcoin library or set of libraries, which will have very robust and very strong abstractions for dealing with Bitcoin in Rust and we will export this library from our code base perhaps as one of the packages in the workspace and this should also be 100% unit tested, and it should also adhere to the idea of pure core imperative shell.
